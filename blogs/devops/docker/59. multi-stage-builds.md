# Multi-Stage Builds

## Overview

Multi-stage builds are a powerful feature in Docker that allow you to create multiple intermediate containers, copying only the necessary artifacts to the final image.

---

## Problem Statement

How do you reduce the size of your Docker images while keeping build dependencies separate from runtime dependencies? Multi-stage builds solve this efficiently.

---

## Main Content Section 1

### Why Multi-Stage Builds?

Traditional Docker builds include everything in the final image:
* Build tools and compilers
* Source code
* Intermediate files

Multi-stage builds let you:
* Build in one stage
* Copy only what's needed to the final stage
* Reduce image size significantly

---

## Main Content Section 2

### How It Works

Each stage uses a different base image, and you selectively copy artifacts between stages.

---

## Code Example

```dockerfile
# Build stage
FROM node:18 AS builder
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
RUN npm run build

# Production stage
FROM node:18-alpine
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
EXPOSE 3000
CMD ["node", "dist/index.js"]
```

---

## Best Practices

* Use alpine-based images for final stage
* Copy only necessary files
* Use build arguments for flexibility
* Name your stages for clarity

---

## Common Mistakes

* Copying unnecessary files
* Not using .dockerignore
* Keeping build tools in final image
* Not optimizing layer caching

---

## Summary

Multi-stage builds significantly reduce image size and improve security by excluding build-time dependencies from production images.
