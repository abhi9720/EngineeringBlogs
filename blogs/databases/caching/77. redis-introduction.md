# Redis: The Complete Guide

## Overview

Redis (Remote Dictionary Server) is an in-memory data structure store that serves as a database, cache, and message broker. It supports various data structures including strings, hashes, lists, sets, sorted sets, bitmaps, and more. Redis is known for its exceptional performance—operations are typically executed in less than a millisecond—and its versatility in solving diverse caching and real-time problems.

Originally created by Salvatore Sanfilippo in 2009, Redis has become a fundamental piece of modern application architecture, used by companies like Twitter, GitHub, Stack Overflow, and Airbnb.

---

## Why Redis Matters

### The Performance Advantage

Redis stores data in memory, providing read and write speeds impossible for disk-based databases:

| Operation | Typical Latency |
|-----------|-----------------|
| Redis | ~0.1-1ms |
| Memory-mapped | ~1-10ms |
| SSD database | ~5-20ms |
| HDD database | ~10-100ms |

This speed difference makes Redis invaluable for:
- Caching frequently accessed data
- Session management
- Real-time features
- Rate limiting
- Message queues

## Core Data Structures

### 1. Strings

The most basic type—can store strings, numbers, or even serialized objects.

```bash
# Basic operations
SET user:123:name "John Doe"
GET user:123:name

# With expiration (TTL)
SET session:abc "data" EX 3600  # Expires in 3600 seconds (1 hour)
SETNX lock:resource "1"  # SET if Not eXists - for distributed locks

# Numeric operations (perfect for counters)
INCR page:views:home
DECR inventory:product:123
INCRBY users:count 100
```

**Use cases**: Session data, counters, caching serialized objects.

```python
# Python example: Store session data
import redis

r = redis.Redis(host='localhost', port=6379)

# Store session with 30-minute TTL
session_data = {'user_id': 123, 'name': 'John', 'cart_items': 5}
r.setex(f"session:{session_id}", 1800, json.dumps(session_data))

# Retrieve session
stored = r.get(f"session:{session_id}")
session = json.loads(stored) if stored else None
```

### 2. Hashes

Perfect for storing objects—field-value pairs within a single key.

```bash
# Hash operations
HSET user:123 name "John Doe" email "john@example.com" age "30"
HGET user:123 name
HGETALL user:123

# Individual field updates
HSET user:123 phone "555-1234"
HINCRBY user:123 login_count 1

# Check if field exists
HEXISTS user:123 email
```

**Use cases**: User profiles, storing objects without serialization.

```python
# Store user profile
r.hset("user:123", mapping={
    "name": "John Doe",
    "email": "john@example.com",
    "signup_date": "2024-01-15"
})

# Get all user data
user_data = r.hgetall("user:123")
```

### 3. Lists

Ordered collections of strings—perfect for queues and activity feeds.

```bash
# List operations
RPUSH queue:emails "email:1" "email:2" "email:3"  # Add to right
LPOP queue:emails  # Remove from left (FIFO queue!)

# Useful for recent items lists
LPUSH user:123:feed "post:456"  # Add new item to front
LRANGE user:123:feed 0 9  # Get 10 most recent items
```

**Use cases**: Message queues, task processing, recent activity logs.

```python
# Task queue pattern
def enqueue_task(task_data):
    r.rpush("tasks:queue", json.dumps(task_data))

def process_task():
    task = r.lpop("tasks:queue")
    if task:
        process(json.loads(task))
    else:
        time.sleep(1)  # Wait for more tasks
```

### 4. Sets

Unordered collections of unique elements—great for tags, relationships, and deduplication.

```bash
# Set operations
SADD tags:article:123 "redis" "database" "caching"
SMEMBERS tags:article:123

# Check membership
SISMEMBER tags:article:123 "redis"

# Set operations
SADD user:123:friends "456" "789" "101"
SADD user:456:friends "123" "789"
SINTER user:123:friends user:456:friends  # Common friends
SUNION user:123:friends user:456:friends  # All friends
```

**Use cases**: Friend connections, unique visitors, tags, social features.

```python
# Track unique visitors
def track_visitor(page_id, visitor_id):
    r.sadd(f"visitors:{page_id}", visitor_id)

def get_unique_visitor_count(page_id):
    return r.scard(f"visitors:{page_id}")
```

### 5. Sorted Sets

Sets with associated scores—perfect for leaderboards, rankings, and priority queues.

```bash
# Sorted set operations
ZADD leaderboard:game 100 "player1" 200 "player2" 150 "player3"

# Get top players
ZREVRANGE leaderboard:game 0 9 WITHSCORES

# Get player's rank
ZRANK leaderboard:game "player2"  # 1 (0-indexed)
ZREVRANK leaderboard:game "player2"  # 0 (highest first)

# Range queries
ZRANGEBYSCORE leaderboard:game 100 200
```

**Use cases**: Leaderboards, rankings, priority queues, time-series data.

```python
# Leaderboard implementation
def update_score(player_id, points):
    r.zincrby("leaderboard:game", points, player_id)

def get_top_players(limit=10):
    return r.zrevrange("leaderboard:game", 0, limit - 1, withscores=True)

def get_player_rank(player_id):
    return r.zrevrank("leaderboard:game", player_id)
```

## Common Use Cases

### 1. Caching

The most common use case—cache expensive database queries or computations.

```python
def get_user_profile(user_id):
    # Try cache first
    cached = r.get(f"user:{user_id}:profile")
    if cached:
        return json.loads(cached)
    
    # Query database
    user = database.query("SELECT * FROM users WHERE id = ?", user_id)
    
    # Cache for 1 hour
    if user:
        r.setex(f"user:{user_id}:profile", 3600, json.dumps(user))
    
    return user
```

### 2. Session Store

Fast session management for web applications.

```python
# Create session
session_id = generate_secure_id()
r.setex(f"session:{session_id}", 86400, json.dumps({
    "user_id": user_id,
    "login_time": time.time()
}))
```

### 3. Rate Limiting

Protect your API from abuse.

```python
def is_rate_limited(client_id, max_requests=100, window_seconds=60):
    key = f"ratelimit:{client_id}"
    current = r.incr(key)
    
    if current == 1:
        r.expire(key, window_seconds)
    
    return current > max_requests
```

### 4. Pub/Sub

Real-time messaging between components.

```python
# Publisher
r.publish("notifications:user:123", json.dumps({
    "type": "new_message",
    "from": "user:456"
}))

# Subscriber (runs in separate process/thread)
pubsub = r.pubsub()
pubsub.subscribe("notifications:user:123")
for message in pubsub.listen():
    if message['type'] == 'message':
        handle_notification(json.loads(message['data']))
```

### 5. Distributed Locks

Coordinate access to shared resources.

```python
def acquire_lock(lock_name, timeout=10):
    result = r.set(f"lock:{lock_name}", "1", nx=True, ex=timeout)
    return bool(result)

def release_lock(lock_name):
    r.delete(f"lock:{lock_name}")

# Usage
if acquire_lock("process:data"):
    try:
        process_data()
    finally:
        release_lock("process:data")
```

## Redis Persistence

Redis provides two persistence options:

### RDB (Redis Database)

Periodic snapshots—creates point-in-time snapshots.

```bash
# Configuration (redis.conf)
save 900 1    # After 1 change in 900 seconds
save 300 10   # After 10 changes in 300 seconds
save 60 10000 # After 10000 changes in 60 seconds
```

**Pros**: Compact, fast restoration
**Cons**: Potential data loss between snapshots

### AOF (Append-Only File)

Logs every write operation—replays to restore.

```bash
# Configuration
appendonly yes
appendfsync everysec  # everysec is good balance
```

**Pros**: More durable, append-only
**Cons**: Larger files, slightly slower

### Production Recommendation

```bash
# For most cases, use both:
# AOF for durability
# RDB for background snapshots/backups

appendonly yes
appendfsync everysec
save 900 1
save 300 10
save 60 10000
```

## Redis in Production

### Redis Sentinel

For high availability—automatic failover.

```
┌─────────────┐
│  Sentinel   │  (monitors)
└──────┬──────┘
       │
   ┌───┴───┐
   │       │
┌──┴──┐ ┌──┴──┐
│Master│ │Replica│
└──────┘ └──────┘
```

Sentinel monitors master and replicas, promotes replica if master fails, redirects clients.

### Redis Cluster

For horizontal scaling and high availability.

```
┌─────────────┐
│ Redis Cluster│
│ (automated │
│ partitioning)│
└──────┬──────┘
       │
   ┌───┴───┐
   │       │
┌──┴──┐ ┌──┴──┐
│Slot │ │Slot │
│0-5460│ │5461-│
└──────┘ │16383│
         └─────┘
```

Key features:
- Automatic sharding across nodes
- Replication for high availability
- No single point of failure

## Best Practices

### 1. Use Appropriate TTLs

```python
# BAD: No expiration - memory leak risk
r.set("data:temp", value)

# GOOD: Set reasonable TTL
r.setex("data:temp", 3600, value)  # 1 hour
```

### 2. Choose the Right Data Structure

```python
# BAD: Store list as serialized JSON string
r.set("users:recent", json.dumps([1,2,3]))

# GOOD: Use list for collections
r.rpush("users:recent", 1, 2, 3)
```

### 3. Use Connection Pooling

```python
# BAD: Create new connection each time
r = redis.Redis()

# GOOD: Use connection pool
pool = redis.ConnectionPool(host='localhost', port=6379, max_connections=50)
r = redis.Redis(connection_pool=pool)
```

### 4. Monitor Memory Usage

```bash
# Check memory usage
INFO memory

# Find big keys
redis-cli --bigkeys
```

### 5. Be Careful with KEYS and SCAN

```bash
# BAD: KEYS blocks the server
KEYS pattern

# GOOD: Use SCAN for large datasets
SCAN 0 MATCH pattern COUNT 1000
```

## Common Mistakes

### 1. Using Redis as Primary Database

```sql
-- DON'T use Redis as your only data store
-- It's in-memory - data can be lost!
-- Use for caching, not persistence
```

### 2. Not Setting Expiration

```sql
-- DON'T forget TTL on cache keys
-- Data will accumulate forever
-- Run out of memory eventually
```

### 3. Storing Large Values

```sql
-- DON'T store large objects
-- Keep values under 1MB ideally
-- Split large data into chunks if needed
```

### 4. Ignoring Serialization Overhead

```sql
-- Consider JSON vs MessagePack vs Pickle
-- For performance, use more compact formats
-- Or store as hash/list instead of JSON
```

## Summary

Redis is an essential tool in modern application architecture:

- **Fast**: In-memory storage provides sub-millisecond access
- **Versatile**: Multiple data structures for different needs
- **Reliable**: Persistence and clustering options for production
- **Simple**: Easy to learn, powerful in practice

Common use cases:
- **Caching**: Reduce database load, improve response times
- **Sessions**: Fast session storage
- **Rate limiting**: Protect APIs
- **Pub/Sub**: Real-time messaging
