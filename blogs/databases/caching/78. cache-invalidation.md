# Cache Invalidation

## Overview

Cache invalidation is the process of removing or updating cached data when the underlying data changes. Invalidating incorrectly leads to stale data; invalidating too often loses the benefits of caching.

---

## Problem Statement

Caches become stale when source data changes but the cache doesn't. Finding the right balance between cache freshness and hit rate determines application reliability and performance.

---

## Main Content Section 1

### Invalidation Strategies

* **TTL (Time-To-Live)**: Expire entries after a fixed time
* **Event-Driven**: Invalidate on data change events
* **Active Invalidation**: Explicitly delete/update cache on writes
* **Version-Based**: Include version in key, invalidate on change
* **Probabilistic**: Random early expiration based on staleness probability

---

## Main Content Section 2

### Write Strategies

* **Cache-Aside**: Write to DB, then invalidate cache
* **Write-Through**: Write to cache and DB together
* **Write-Behind**: Write to cache, async write to DB
* **Refresh-Ahead**: Proactively refresh expiring entries

## Code Example

```python
# Cache-Aside (Read heavy, tolerate occasional staleness)
def get_user(user_id):
    cached = redis.get(f"user:{user_id}")
    if cached:
        return json.loads(cached)
    user = db.query("SELECT * FROM users WHERE id = ?", user_id)
    redis.setex(f"user:{user_id}", 300, json.dumps(user))  # 5 min TTL
    return user

def update_user(user_id, data):
    db.execute("UPDATE users SET ... WHERE id = ?", user_id)
    redis.delete(f"user:{user_id}")  # Invalidate cache

# Version-based invalidation
current_version = db.get_version("user", user_id)  # e.g., v15
cache_key = f"user:{user_id}:v{current_version}"
# If version changes, old cache keys are automatically ignored
```

---

## Best Practices

* Choose TTL based on data volatility
* Use event sourcing for reliable invalidation
* Implement graceful fallback to database
* Log cache invalidation for debugging
* Consider consistency requirements per data type

---

## Common Mistakes

* Forgetting to invalidate on writes
* Setting TTL too long for frequently changing data
* Creating cache stampedes on expiration
* Not handling partial invalidation failures
* Over-invalidation reducing cache hit rate

---

## Summary

Cache invalidation requires balancing data freshness with cache efficiency. Choose strategies based on your data's change frequency and your application's consistency requirements.
