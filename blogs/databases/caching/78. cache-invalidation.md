# Cache Invalidation: Strategies and Best Practices

## Overview

Cache invalidation is often considered one of the hardest problems in computer science—and for good reason. When data changes in your primary database, how do you ensure the cached data reflects those changes? Getting this wrong leads to stale data, user confusion, or lost revenue. Getting it right requires understanding the trade-offs between data freshness, cache efficiency, and system complexity.

This guide covers the major invalidation strategies, when to use each, and how to implement them in practice.

---

## The Cache Invalidation Problem

### Why It's Hard

```python
# Timeline showing the problem:
# Time T0: User A reads data, cache is populated
# Time T1: User B updates the data in the database
# Time T2: User A reads from cache, gets STALE data!

# This window of staleness is the core problem
```

The fundamental issue: **caches are snapshots of data at a point in time, but data changes over time**.

The solution must balance:
- **Freshness**: How current is the cached data?
- **Efficiency**: How often can we serve from cache vs. database?
- **Complexity**: How hard is the solution to implement and maintain?

## Invalidation Strategies

### 1. TTL (Time-To-Live)

The simplest approach: every cached item has an expiration time.

```python
def get_user(user_id):
    # Try cache first
    cached = redis.get(f"user:{user_id}")
    if cached:
        return json.loads(cached)
    
    # Cache miss - fetch from database
    user = db.query("SELECT * FROM users WHERE id = ?", user_id)
    
    # Cache with TTL (5 minutes)
    redis.setex(f"user:{user_id}", 300, json.dumps(user))
    
    return user
```

**When to use**:
- Data that doesn't need to be 100% current
- Read-heavy workloads
- When occasional staleness is acceptable

**Trade-offs**:
- Simple to implement
- May serve stale data until TTL expires
- TTL too short = low hit rate
- TTL too long = stale data

**Setting TTL by data type**:

```python
# Highly volatile data (e.g., inventory counts)
ttl = 30  # seconds

# User profile data
ttl = 300  # 5 minutes

# Reference data (e.g., product categories)
ttl = 3600  # 1 hour

# Static configuration
ttl = 86400  # 24 hours
```

### 2. Event-Driven Invalidation

Invalidate cache when data changes, using events or triggers.

```python
# When updating user, publish event
def update_user(user_id, data):
    db.execute("UPDATE users SET ... WHERE id = ?", user_id)
    
    # Publish invalidation event
    redis.publish(f"cache:invalidate:user:{user_id}", "invalidate")

# Consumer handles invalidation
pubsub = redis.pubsub()
pubsub.subscribe("cache:invalidate:user:*")

for message in pubsub.listen():
    if message['type'] == 'message':
        # Delete related cache entries
        user_id = message['channel'].split(':')[-1]
        redis.delete(f"user:{user_id}")
```

**When to use**:
- Need immediate invalidation on changes
- Multiple services share a cache
- Complex invalidation logic required

**Trade-offs**:
- Immediate consistency
- More complex infrastructure
- Event delivery isn't guaranteed

### 3. Active Invalidation on Write

When data changes, explicitly delete or update the cache immediately.

```python
def update_user(user_id, data):
    # Update database
    db.execute("UPDATE users SET ... WHERE id = ?", user_id)
    
    # Invalidate cache immediately
    redis.delete(f"user:{user_id}")
    
    # Optional: Warm cache immediately
    user = db.query("SELECT * FROM users WHERE id = ?", user_id)
    redis.setex(f"user:{user_id}", 300, json.dumps(user))

# Alternative: Update cache directly (if cache is authoritative for reads)
def update_user_direct(user_id, data):
    # Update cache directly (bypasses database - risky!)
    redis.hset(f"user:{user_id}", mapping=data)
```

**When to use**:
- Data must be fresh
- When you control both read and write paths

**Trade-offs**:
- Ensures freshness
- Two operations (write + delete) must be atomic
- Cache might be temporarily inconsistent

### 4. Version-Based Invalidation

Include version information in the cache key—when data changes, version changes, making old keys effectively invalid.

```python
def get_user(user_id):
    # Get current version
    version = redis.get(f"user:{user_id}:version")
    
    if not version:
        # No data, fetch and cache
        user = db.query("SELECT * FROM users WHERE id = ?", user_id)
        version = user['version']
        key = f"user:{user_id}:v{version}"
        redis.setex(key, 300, json.dumps(user))
        redis.setex(f"user:{user_id}:version", 86400, version)
        return user
    
    # Check if cached version matches current version
    key = f"user:{user_id}:v{version}"
    cached = redis.get(key)
    
    if cached:
        return json.loads(cached)
    
    # Version changed, fetch new data
    user = db.query("SELECT * FROM users WHERE id = ?", user_id)
    key = f"user:{user_id}:v{user['version']}"
    redis.setex(key, 300, json.dumps(user))
    return user
```

**When to use**:
- Multiple related cache entries
- When version changes are rare but important
- Need fine-grained invalidation

**Trade-offs**:
- No explicit delete needed
- More complex key management
- Version tracking adds overhead

### 5. Write-Through and Write-Behind

These are actually write strategies that affect cache consistency (covered in detail in the "Write-Through vs Write-Back" blog):

- **Write-Through**: Write to both cache and database together
- **Write-Behind**: Write to cache first, then async to database

## Handling the Cache Stampede Problem

When a cached item expires, multiple requests might simultaneously try to rebuild the cache, overwhelming the database.

### The Problem

```python
# At T0, cache expires
# At T0+0.001s: 100 requests hit cache miss
# At T0+0.002s: All 100 try to query database
# Database overwhelmed!
```

### Solutions

**1. Lock-based approach**:

```python
def get_user(user_id):
    cached = redis.get(f"user:{user_id}")
    if cached:
        return json.loads(cached)
    
    # Try to acquire lock
    lock = redis.lock(f"user:{user_id}:lock", timeout=10)
    if lock.acquire():
        try:
            # Double-check after acquiring lock
            cached = redis.get(f"user:{user_id}")
            if cached:
                return json.loads(cached)
            
            # Fetch from database
            user = db.query("SELECT * FROM users WHERE id = ?", user_id)
            redis.setex(f"user:{user_id}", 300, json.dumps(user))
            return user
        finally:
            lock.release()
    else:
        # Wait for other request to populate cache
        time.sleep(0.1)
        return get_user(user_id)  # Retry
```

**2. Probabilistic early expiration**:

```python
def get_user(user_id):
    cached = redis.get(f"user:{user_id}")
    
    # Add some randomness to refresh probability
    if cached:
        data = json.loads(cached)
        age = redis.ttl(f"user:{user_id}")  # Time until expiry
        
        # If nearly expired (less than 10% of TTL remaining) and 
        # random chance hits, refresh proactively
        if age < 30 and random.random() < 0.1:
            # Async refresh (fire and forget)
            thread = threading.Thread(target=refresh_user_cache, args=(user_id,))
            thread.start()
        
        return data
    
    # Cache miss - fetch from database
    user = db.query("SELECT * FROM users WHERE id = ?", user_id)
    redis.setex(f"user:{user_id}", 300, json.dumps(user))
    return user
```

**3. Request queuing**:

```python
# Use a task queue to handle cache misses
def get_user_async(user_id):
    cached = redis.get(f"user:{user_id}")
    if cached:
        return json.loads(cached)
    
    # Check if there's already a pending request
    if not redis.set(f"user:{user_id}:pending", "1", nx=True, ex=5):
        # Someone else is fetching, wait a bit and retry
        time.sleep(0.05)
        return get_user_async(user_id)
    
    # We need to fetch - use task queue
    queue.enqueue("fetch_user", user_id)
    
    # Wait for result (or return stale if acceptable)
    for _ in range(10):  # 1 second max wait
        cached = redis.get(f"user:{user_id}")
        if cached:
            return json.loads(cached)
        time.sleep(0.1)
    
    # Fallback to database
    return db.query("SELECT * FROM users WHERE id = ?", user_id)
```

## Best Practices

### 1. Choose TTL Based on Data Volatility

```python
# Order of volatility (most to least volatile):
# 1. Real-time data (stock prices): 0-30 seconds
# 2. User-generated content: 30-300 seconds
# 3. User profiles: 300-3600 seconds
# 4. Reference data: 3600+ seconds
```

### 2. Design for Invalidation

```python
# Design cache keys to make invalidation easy
# GOOD: user:123:profile, user:123:orders, user:123:preferences
# Makes it easy to invalidate all user-related cache

# BAD: profile_for_user_123
# More difficult to find and invalidate
```

### 3. Use Multi-Level Invalidation When Needed

```python
# When user updates profile:
# 1. Update database
# 2. Invalidate user profile cache
# 3. Publish event for other services to invalidate their caches
# 4. Update any aggregate caches (e.g., "users with most followers")
```

### 4. Log and Monitor Invalidation

```python
def invalidate_user(user_id, reason):
    # Log invalidation
    logger.info(f"Cache invalidated for user:{user_id}, reason: {reason}")
    
    # Metrics
    metrics.increment("cache.invalidations")
    
    redis.delete(f"user:{user_id}")
```

### 5. Consider Consistency Requirements Per Data Type

```python
# These need strong consistency (immediate invalidation):
# - Account balances
# - Inventory counts  
# - Payment status

# These can tolerate eventual consistency (TTL-based):
# - User profile details
# - Product descriptions
# - Search results
```

## Common Mistakes

### 1. Forgetting to Invalidate on Writes

```python
# BAD: Update database but not cache
def update_user(user_id, data):
    db.execute("UPDATE users SET ...", user_id, data)
    # Forgot to invalidate cache!
    # Users will see stale data until TTL expires
```

### 2. Setting TTL Too Long for Volatile Data

```python
# BAD: 24-hour TTL on inventory counts
# If inventory sells out, users see wrong data for hours!
```

### 3. Not Handling Invalidation Failures

```python
# BAD: Assume invalidation always succeeds
redis.delete(f"user:{user_id}")  # What if this fails?

# GOOD: Add retry logic
for attempt in range(3):
    try:
        redis.delete(f"user:{user_id}")
        break
    except Exception as e:
        logger.warning(f"Invalidation failed: {e}")
        time.sleep(0.1)
```

### 4. Over-Invalidation

```python
# BAD: Invalidate entire cache when just one item changes
def update_product(product_id):
    # Update database
    db.execute("UPDATE products SET ...", product_id)
    
    # Invalidate everything - too broad!
    redis.delete("products:all")  
    redis.flushdb()  # Really bad!
```

### 5. Ignoring Cache Stampedes

```python
# The "thundering herd" problem described earlier
# Multiple simultaneous requests trying to rebuild cache
```

## Summary

Cache invalidation is critical for maintaining data consistency:

- **TTL-based**: Simple but may serve stale data
- **Event-driven**: Immediate but complex
- **Active invalidation**: Direct but requires careful coordination
- **Version-based**: Robust for complex scenarios

Key principles:
1. Choose strategy based on data consistency requirements
2. Consider cache stampedes and handle them
3. Log and monitor invalidation
4. Test your invalidation logic
5. Balance freshness with hit rate
