# Write-Through vs Write-Back: Complete Comparison

## Overview

When implementing caching in your application, you must decide how to handle writes. The two primary strategies are **write-through** (synchronous writes to cache and database) and **write-back** (async writes to database after cache updates). This choice fundamentally affects your application's consistency guarantees, performance characteristics, and complexity.

This guide provides a comprehensive comparison to help you choose the right strategy for your use case.

---

## The Write Problem

When you cache data, you create a secondary copy. The challenge is: what happens when the source data changes?

```
┌─────────────────────────────────────────────────────────────┐
│                   THE CACHE CONSISTENCY PROBLEM             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   Time T0: Application writes data                          │
│            ┌─────────┐                                      │
│            │   App   │ ─────► DB                            │
│            └─────────┘                                      │
│                                                             │
│   Time T0+1: Should we write to cache?                      │
│            ┌─────────┐                                      │
│            │  Cache  │ ◄─── What happens here?             │
│            └─────────┘                                      │
│                                                             │
│   Options:                                                  │
│   1. Write-through: Write to both together                  │
│   2. Write-back: Write to cache, async to DB               │
│   3. Write-around: Write to DB only                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

---

## Write-Through

### How It Works

Write-through ensures data is written to both the cache and the database synchronously before considering the operation complete.

```
┌─────────────────────────────────────────────────────────────┐
│                    WRITE-THROUGH FLOW                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   Application                                               │
│       │                                                     │
│       ▼                                                     │
│   ┌─────────────┐                                           │
│   │Write to     │                                            │
│   │cache        │                                            │
│   └──────┬──────┘                                           │
│          │ ✓                                                │
│          ▼                                                  │
│   ┌─────────────┐  (wait for success)                       │
│   │Write to     │                                            │
│   │database     │                                            │
│   └──────┬──────┘                                           │
│          │ ✓                                                │
│          ▼                                                  │
│   Success!  Both cache and DB are in sync                   │
│                                                             │
│   Properties:                                               │
│   - Synchronous: Both writes must complete                  │
│   - Consistent: Cache always has latest data                │
│   - Higher latency: Two writes = more time                   │
│   - Safer: No risk of losing data                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Implementation

```python
import threading

class WriteThroughCache:
    def __init__(self, cache, db):
        self.cache = cache
        self.db = db
    
    def write_user(self, user_id, data):
        """
        Write-through: Both cache and database are updated,
        and we wait for both to succeed before returning.
        """
        cache_key = f"user:{user_id}"
        
        # Step 1: Write to cache
        self.cache.set(cache_key, data)
        
        # Step 2: Write to database (synchronous)
        self.db.execute(
            "INSERT INTO users (id, name, email) VALUES (?, ?, ?)",
            user_id, data['name'], data['email']
        )
        
        # Both succeeded - data is consistent
        return True
    
    def update_user(self, user_id, data):
        """Update with write-through"""
        cache_key = f"user:{user_id}"
        
        # Write to cache
        self.cache.set(cache_key, data)
        
        # Write to database
        self.db.execute(
            "UPDATE users SET name = ?, email = ? WHERE id = ?",
            data['name'], data['email'], user_id
        )
        
        return True
    
    def delete_user(self, user_id):
        """Delete with write-through"""
        cache_key = f"user:{user_id}"
        
        # Remove from cache
        self.cache.delete(cache_key)
        
        # Remove from database
        self.db.execute("DELETE FROM users WHERE id = ?", user_id)
        
        return True
```

### When to Use Write-Through

- **Financial transactions**: Where data loss is unacceptable
- **Inventory management**: Accurate counts are critical
- **User authentication**: Critical security data
- **Configuration data**: Must be consistent across restarts
- **Any scenario requiring strong consistency**

### Trade-offs

**Advantages**:
- **Strong consistency**: Cache always has the latest data
- **No data loss risk**: Data persisted in both places
- **Simple reasoning**: Cache is always accurate
- **Read-after-write**: Can immediately read what you just wrote

**Disadvantages**:
- **Higher write latency**: Two writes take longer than one
- **Lower throughput**: More operations per write
- **Cache overhead**: Cache must handle writes in real-time
- **Reduced benefit for write-heavy workloads**: No advantage over direct DB writes

## Write-Back (Write-Behind)

### How It Works

Write-back buffers writes to the cache and asynchronously writes to the database. The application continues immediately after writing to cache.

```
┌─────────────────────────────────────────────────────────────┐
│                     WRITE-BACK FLOW                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   Application                                               │
│       │                                                     │
│       ▼                                                     │
│   ┌─────────────┐                                           │
│   │Write to     │                                            │
│   │cache        │                                            │
│   └──────┬──────┘                                           │
│          │ ✓ (immediate!)                                  │
│          ▼                                                  │
│   Success! (Returns immediately)                            │
│          │                                                  │
│          ▼ (async, in background)                          │
│   ┌─────────────┐                                           │
│   │Queue write  │  ──► Database (eventually)               │
│   │for DB       │                                            │
│   └─────────────┘                                           │
│                                                             │
│   Properties:                                               │
│   - Asynchronous: DB write is deferred                     │
│   - Eventually consistent: Cache leads DB                   │
│   - Lower latency: Returns immediately after cache write     │
│   - Risk: Data loss if cache fails before DB flush          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Implementation

```python
import threading
import time

class WriteBackCache:
    def __init__(self, cache, db, flush_interval=5):
        self.cache = cache
        self.db = db
        self.pending_writes = []  # Queue of pending writes
        self.lock = threading.Lock()
        self.running = True
        
        # Background thread for flushing
        self.flusher = threading.Thread(target=self._flush_loop)
        self.flusher.daemon = True
        self.flusher.start()
    
    def _flush_loop(self):
        """Background thread that periodically flushes pending writes"""
        while self.running:
            time.sleep(5)  # Flush every 5 seconds
            self._flush_pending()
    
    def _flush_pending(self):
        """Flush all pending writes to database"""
        with self.lock:
            if not self.pending_writes:
                return
            
            # Take all pending writes
            writes_to_flush = self.pending_writes
            self.pending_writes = []
        
        # Attempt to flush to database
        failed = []
        for key, value, operation in writes_to_flush:
            try:
                if operation == 'write':
                    self.db.set(key, value)
                elif operation == 'delete':
                    self.db.delete(key)
            except Exception as e:
                print(f"Failed to flush {key}: {e}")
                failed.append((key, value, operation))
        
        # Re-queue failed writes
        if failed:
            with self.lock:
                self.pending_writes.extend(failed) + self.pending_writes
    
    def write_user(self, user_id, data):
        """Write-back: Update cache immediately, queue DB write"""
        cache_key = f"user:{user_id}"
        
        # Step 1: Write to cache (fast)
        self.cache.set(cache_key, data)
        
        # Step 2: Queue for async database write
        with self.lock:
            self.pending_writes.append((cache_key, data, 'write'))
        
        # Return immediately (don't wait for DB)
        return True
    
    def delete_user(self, user_id):
        """Delete with write-back"""
        cache_key = f"user:{user_id}"
        
        # Remove from cache
        self.cache.delete(cache_key)
        
        # Queue for database delete
        with self.lock:
            self.pending_writes.append((cache_key, None, 'delete'))
        
        return True
    
    def shutdown(self):
        """Flush pending writes before shutdown"""
        self.running = False
        self.flusher.join()
        self._flush_pending()  # Final flush
```

### When to Use Write-Back

- **High-volume writes**: Where throughput matters more than immediate consistency
- **Analytics and metrics**: Aggregated data that can tolerate some loss
- **Logging and events**: Where occasional missed events are acceptable
- **Time-series data**: High-frequency sensor data
- **Non-critical operations**: Data that can be recomputed if lost

### Trade-offs

**Advantages**:
- **Lower write latency**: Application doesn't wait for database
- **Higher throughput**: Can batch multiple writes
- **Write coalescing**: Multiple updates can be combined
- **Better performance**: Optimizes for speed

**Disadvantages**:
- **Data loss risk**: If cache fails before flush, data is lost
- **Eventual consistency**: Cache and DB may be temporarily out of sync
- **Complexity**: Need to handle queue, failures, and ordering
- **Difficult debugging**: Hard to track what was persisted

## Comparison Summary

| Aspect | Write-Through | Write-Back |
|--------|---------------|------------|
| **Consistency** | Strong (immediate) | Eventual |
| **Write Latency** | Higher (2 writes) | Lower (1 write) |
| **Throughput** | Lower | Higher |
| **Data Loss Risk** | None | Possible |
| **Complexity** | Simple | More complex |
| **Read-after-write** | Yes | May return stale |
| **Use Case** | Critical data | High-volume, non-critical |

### Write-Through + Write-Behind Hybrid

```python
class HybridCache:
    """
    Use write-through for critical data,
    write-back for high-volume data
    """
    
    CRITICAL_KEYS = {'user:', 'order:', 'payment:'}
    
    def __init__(self, cache, db):
        self.cache = cache
        self.db = db
    
    def write(self, key, value):
        if self._is_critical(key):
            return self._write_through(key, value)
        else:
            return self._write_back(key, value)
    
    def _is_critical(self, key):
        return any(key.startswith(prefix) for prefix in self.CRITICAL_KEYS)
    
    def _write_through(self, key, value):
        # Critical: Write both synchronously
        self.cache.set(key, value)
        self.db.set(key, value)
        return True
    
    def _write_back(self, key, value):
        # Non-critical: Write cache immediately, async DB
        self.cache.set(key, value)
        self._queue_for_db(key, value)
        return True
```

## Implementation Best Practices

### For Write-Through

```python
# Use transactions to ensure both writes succeed or fail together
def write_with_transaction(cache, db, key, value):
    try:
        db.begin()
        
        cache.set(key, value)
        db.set(key, value)
        
        db.commit()
        return True
    except Exception:
        db.rollback()
        # Rollback cache too
        cache.delete(key)
        raise
```

```python
def get_queue_stats(self):
```

## Common Mistakes

### Using Write-Back for Critical Data

```python
# WRONG: Using write-back for financial transactions
# If cache crashes, you could lose payments!
def process_payment(user_id, amount):
    cache.set(f"balance:{user_id}", new_balance)
    # Not flushed to DB yet!
    return "Payment processed"  # User thinks it's done
```

### Not Handling Flush Failures

```python
# WRONG: No retry logic for failed writes
def _flush_pending(self):
    for key, value in self.pending_writes:
        try:
            self.db.set(key, value)
        except:
            # Lost data!
            pass
```

### Ignoring Write Coalescing

```python
# For high-frequency writes, batch them
# Instead of writing each update immediately:
# 100 writes per second → 100 DB writes

# Batch updates:
# 100 writes per second → 10 DB writes (every 0.1s)
```

### Not Flushing on Application Shutdown

```python
# WRONG: App crashes, pending writes lost
app = WriteBackCache(...)
app.run()
# Crash! All pending writes lost!

# CORRECT: Graceful shutdown
def signal_handler(signum, frame):
    app.shutdown()
    sys.exit(0)
```

## Summary

Both write-through and write-back have their place:

- **Write-Through**: For critical data requiring strong consistency
  - Strong consistency guarantee
  - Higher latency but safer
  - Use for: financial data, user profiles, inventory

- **Write-Back**: For high-throughput, eventually consistent scenarios
  - Lower latency, higher throughput
  - Risk of data loss
  - Use for: analytics, logging, metrics

- **Hybrid**: Use both strategically for different data types

Key takeaways:
1. Match strategy to data criticality
2. Monitor write-back queue depth
3. Implement proper failure handling
4. Flush on shutdown
5. Consider eventual consistency needs
