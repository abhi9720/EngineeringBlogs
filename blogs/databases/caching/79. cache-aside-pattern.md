# Cache-Aside Pattern

## Overview

Cache-aside (lazy loading) is the most common caching pattern. The application manages the cache explicitly: checking the cache first, loading from the database on miss, and updating/invalidating on writes.

---

## Problem Statement

Applications need fast reads without always hitting the database. Reading directly from cache and lazily populating it on cache misses provides both speed and efficient cache utilization.

---

## Main Content Section 1

### How It Works

* **Read Path**: Check cache → hit return data, miss → query DB → store in cache → return data
* **Write Path**: Update database → invalidate/delete cache entry
* **Key Discovery**: Only cache data that's actually requested

---

## Main Content Section 2

### Pattern Flow

```
READ:
  Client → Check Cache
           ↓ Found? → Return Data
           ↓ Not Found → Query Database
                          → Store in Cache
                          → Return Data

WRITE:
  Client → Update Database
           → Invalidate Cache
```

## Code Example

```python
class UserRepository:
    def __init__(self, db, cache):
        self.db = db
        self.cache = cache

    def get_user(self, user_id):
        cache_key = f"user:{user_id}"
        cached = self.cache.get(cache_key)
        if cached:
            return User.from_json(cached)
        
        user = self.db.query(
            "SELECT * FROM users WHERE id = ?", user_id
        )
        if user:
            self.cache.setex(cache_key, 3600, user.to_json())
        return user

    def update_user(self, user_id, data):
        self.db.execute(
            "UPDATE users SET name = ?, email = ? WHERE id = ?",
            data['name'], data['email'], user_id
        )
        self.cache.delete(f"user:{user_id}")

    def delete_user(self, user_id):
        self.db.execute("DELETE FROM users WHERE id = ?", user_id)
        self.cache.delete(f"user:{user_id}")
```

## Best Practices

* Use consistent key naming conventions
* Set appropriate TTL values
* Handle cache failures gracefully (fallback to DB)
* Implement retry logic for cache operations
* Consider cache stampede prevention

---

## Common Mistakes

* Not invalidating cache on writes
* Forgetting error handling for cache operations
* Using cache as primary storage
* Setting TTL too long for frequently updated data
* Cache stampede on popular keys during cold start

---

## Summary

Cache-aside provides an elegant separation between caching logic and business logic. It offers flexibility, efficient cache utilization, and graceful degradation when cache is unavailable.
