# MongoDB Schema Design

## Overview

MongoDB's flexible schema requires thoughtful design decisions. Unlike relational databases, MongoDB allows you to model data in ways that optimize for your application's access patterns.

---

## Problem Statement

Design choices in MongoDB have significant performance implications. Embedding vs referencing, indexing strategy, and document structure directly affect query speed and write throughput.

---

## Main Content Section 1

### Embedding vs Referencing

**Embedding** (store related data in same document):
* Good for: one-to-few, data accessed together
* Faster reads, single query
* Limited by 16MB document size

**Referencing** (store IDs and query separately):
* Good for: one-to-many, many-to-many
* Reduces document size
* More flexible, requires additional queries

---

## Main Content Section 2

### Schema Design Patterns

* **Attribute Pattern**: Index on varying fields
* **Extended Reference Pattern**: Pre-join frequently accessed data
* **Subset Pattern**: Store only recent/important data in main document
* **Computed Pattern**: Pre-calculate and store results
* **Bucket Pattern**: Group time-series data

## Code Example

```javascript
// Embedding example: Blog post with comments
{
  _id: ObjectId("..."),
  title: "My Post",
  content: "...",
  author: { name: "John", email: "john@example.com" },
  comments: [
    { text: "Great post!", author: "Jane", date: ISODate("...") },
    { text: "Thanks!", author: "John", date: ISODate("...") }
  ]
}

// Referencing example: Orders with separate products
// orders collection
{
  _id: ObjectId("..."),
  customer_id: ObjectId("..."),
  items: [
    { product_id: ObjectId("..."), quantity: 2, price: 19.99 }
  ]
}

// products collection (separate)
{
  _id: ObjectId("..."),
  name: "Widget",
  price: 19.99
}
```

---

## Best Practices

* Design for your application's query patterns
* Use consistent field names across documents
* Consider document growth and update patterns
* Index fields used in queries, sorts, and aggregations
* Avoid unbounded array growth

---

## Common Mistakes

* Over-embedding leading to large documents
* Not planning for future query changes
* Ignoring index cardinality
* Using unbounded arrays
* Neglecting write/update patterns

---

## Summary

MongoDB schema design requires balancing data organization with query performance. Analyze your access patterns and choose embedding or referencing strategically for optimal results.
