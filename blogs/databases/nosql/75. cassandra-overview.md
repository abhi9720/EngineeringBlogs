# Apache Cassandra: A Complete Guide

## Overview

Apache Cassandra is a distributed, masterless, wide-column NoSQL database designed for massive scale and high availability. It powers critical infrastructure at companies like Apple, Netflix, Instagram, and Discord, handling trillions of requests per day with no single point of failure.

Cassandra's architecture combines the best of distributed systems: linear scalability, tunable consistency, and decentralized design. Understanding these fundamentals is essential for anyone building large-scale data systems.

---

## Core Architecture

### Masterless, Distributed Design

Unlike traditional databases with primary-replica architectures, Cassandra has no master node. Every node in a Cassandra cluster is identical and can serve any request. This provides:

- **No single point of failure**: Any node can go down without affecting the system
- **Linear scalability**: Add nodes to increase capacity proportionally
- **Geographic distribution**: Data can be replicated across data centers and regions

```sql
-- Each node is equal in Cassandra
-- Client can connect to any node
-- Node acts as coordinator for the request
```

### The Cluster

A Cassandra cluster is a collection of nodes that work together. Nodes are organized into:

- **Rack**: Group of nodes in the same physical location
- **Data Center**: Collection of racks, typically in the same facility

```
Cluster
├── Data Center 1
│   ├── Rack 1
│   │   ├── Node 1
│   │   └── Node 2
│   └── Rack 2
│       ├── Node 3
│       └── Node 4
└── Data Center 2 (for disaster recovery / global distribution)
    └── ...
```

### Gossip Protocol

Nodes communicate via gossip protocol, which:
- Shares cluster state efficiently
- Detects node failures
- Distributes cluster metadata
- Requires no external coordination

## Data Distribution: Partitioning

### Partition Key

The partition key determines which node stores a row:

```sql
CREATE TABLE users (
    user_id UUID,
    email TEXT,
    name TEXT,
    PRIMARY KEY (user_id)
);
```

Cassandra hashes the partition key to determine which node owns that data. All rows with the same partition key are stored together.

### Composite Partition Keys

For more complex distribution:

```sql
CREATE TABLE orders (
    customer_id UUID,
    order_id TIMEUUID,
    product TEXT,
    amount DECIMAL,
    PRIMARY KEY (customer_id, order_id)
) WITH CLUSTERING ORDER BY (order_id DESC);
```

- **customer_id**: Partition key (determines which node)
- **order_id**: Clustering key (determines sort order within partition)

## Replication and Consistency

### Replication Factor

The replication factor determines how many copies of each row exist:

```sql
CREATE KEYSPACE myapp
WITH REPLICATION = {
    'class': 'SimpleStrategy',
    'replication_factor': 3
};
```

**SimpleStrategy**: For single data center
**NetworkTopologyStrategy**: For multiple data centers (recommended)

```sql
-- Multi-data center recommended approach
CREATE KEYSPACE myapp
WITH REPLICATION = {
    'class': 'NetworkTopologyStrategy',
    'dc1': 3,
    'dc2': 3
};
```

### Tunable Consistency

You can tune consistency per query:

```sql
-- Write with QUORUM (majority of replicas)
CONSISTENCY QUORUM;
INSERT INTO users (user_id, email) VALUES (?, ?);

-- Read with QUORUM
CONSISTENCY QUORUM;
SELECT * FROM users WHERE user_id = ?;

-- Options: ONE, TWO, THREE, QUORUM, ALL, LOCAL_ONE, LOCAL_QUORUM, EACH_QUORUM
```

**Consistency level trade-offs**:
- **ONE**: Fastest, but may read stale data
- **QUORUM**: Balanced (strong consistency with some tolerance)
- **ALL**: Strongest, but slowest (fails if any replica down)

## The Data Model: Queries First

Unlike relational databases, Cassandra data modeling starts with your queries, not your entities.

### Design Process

1. Identify your query patterns
2. Create tables for each query
3. Denormalize freely
4. Optimize for partition design

### Example: From Relational to Cassandra

**Relational approach**: Normalized tables
```sql
-- Normalized: Users, Orders, Products linked by foreign keys
SELECT u.name, o.id, p.name
FROM users u
JOIN orders o ON u.id = o.user_id
JOIN order_items oi ON o.id = oi.order_id
JOIN products p ON oi.product_id = p.id
WHERE u.id = ?;
```

**Cassandra approach**: Query-focused tables
```sql
-- Denormalized: One table per query
CREATE TABLE user_orders (
    user_id UUID,
    order_id TIMEUUID,
    product_name TEXT,
    amount DECIMAL,
    order_date TIMESTAMP,
    PRIMARY KEY (user_id, order_id)
) WITH CLUSTERING ORDER BY (order_id DESC);

-- Query is now simple and efficient:
SELECT * FROM user_orders WHERE user_id = ?;
```

### Anti-Patterns to Avoid

**1. Trying to model relationships**:

```sql
-- BAD: Trying to do joins
SELECT * FROM users u
JOIN orders o ON u.id = o.user_id;

-- Cassandra doesn't support joins! Design for your queries instead.
```

**2. Using too many secondary indexes**:

```sql
-- BAD: Relying on secondary indexes for queries
CREATE INDEX ON users(email);
SELECT * FROM users WHERE email = ?;

-- Secondary indexes are for low-cardinality, not high-cardinality fields
-- Better to create explicit tables if querying by different fields
```

**3. Unbounded partitions**:

```sql
-- BAD: Unbounded partition with too many rows
PRIMARY KEY (user_id);

-- If user has millions of orders, this partition grows too large
-- Better: Use time-based or bucket-based partitioning
PRIMARY KEY ((user_id, month), order_id);
```

## CQL: Cassandra Query Language

CQL looks like SQL but has different semantics:

### Key Differences from SQL

- **No JOINs**: Design your schema around queries instead
- **No GROUP BY**: Use grouping in application or denormalize
- **No OR in WHERE**: Use IN clause or separate queries
- **Limited aggregations**: No joins, count/sum are limited

### Working with CQL

```sql
-- Create keyspace
CREATE KEYSPACE IF NOT EXISTS myapp
WITH REPLICATION = {'class': 'NetworkTopologyStrategy', 'dc1': 3};

-- Create table
CREATE TABLE users (
    user_id UUID PRIMARY KEY,
    email TEXT,
    name TEXT,
    created_at TIMESTAMP
);

-- Insert data
INSERT INTO users (user_id, email, name, created_at)
VALUES (uuid(), 'john@example.com', 'John Doe', now());

-- Query with partition key (efficient!)
SELECT * FROM users WHERE user_id = ?;

-- Collections (be careful with size)
CREATE TABLE user_preferences (
    user_id UUID PRIMARY KEY,
    tags SET<TEXT>,  -- Set of strings
    phone_numbers LIST<TEXT>,  -- Ordered list
    properties MAP<TEXT, TEXT>  -- Key-value
);

INSERT INTO user_preferences (user_id, tags, phone_numbers)
VALUES (?, {'premium', 'beta'}, ['555-1234', '555-5678']);
```

### Practical Data Modeling Examples

**1. Time-series data**:

```sql
-- IoT sensor readings
CREATE TABLE sensor_data (
    sensor_id TEXT,
    reading_time TIMESTAMP,
    temperature DOUBLE,
    humidity DOUBLE,
    PRIMARY KEY (sensor_id, reading_time)
) WITH CLUSTERING ORDER BY (reading_time DESC)
  AND default_time_to_live = 2592000;  -- 30 days TTL
```

**2. User activity tracking**:

```sql
-- Track user actions with bucket
CREATE TABLE user_activity (
    user_id UUID,
    date_bucket TEXT,  -- "2024-01" format
    action_time TIMESTAMP,
    action_type TEXT,
    target_id TEXT,
    PRIMARY KEY ((user_id, date_bucket), action_time)
) WITH CLUSTERING ORDER BY (action_time DESC);

-- Query: Get user's activity for a month
SELECT * FROM user_activity
WHERE user_id = ? AND date_bucket = '2024-01';
```

**3. Shopping cart**:

```sql
-- Cart per user
CREATE TABLE cart (
    user_id UUID PRIMARY KEY,
    items MAP<UUID, INT>,  -- product_id to quantity
    last_updated TIMESTAMP
);

-- Atomic updates
UPDATE cart SET items = items + {? : ?} WHERE user_id = ?;
```

## Performance Considerations

### Writing Data

Cassandra is optimized for writes:

- **Write path**: Data written to commit log → memtable → SSTable
- **No read-before-write**: Updates are append-only
- **Tombstones**: Deleted data marked, not immediately removed

```sql
-- Write is always fast because:
-- 1. Append-only (no seek required)
-- 2. No read before write
-- 3. Memoized in memory first
```

### Reading Data

Reading can be more complex:

```sql
-- Read path: Check memtable, then SSTables (merged)
-- Uses bloom filters to skip irrelevant SSTables
-- May need to merge from multiple SSTables
```

**Optimization techniques**:
- Keep partitions small (avoid hot partitions)
- Use IN efficiently (limited number of keys)
- Limit query results (no "SELECT * FROM table" without WHERE)

### Tombstones

Deleted data leaves tombstones:

```sql
-- When you delete, a tombstone is created
DELETE FROM cart WHERE user_id = ?;

-- Tombstones are removed during compaction
-- Too many tombstones can slow reads!
-- Avoid: Heavy write-then-delete patterns
-- Solution: Use TTL (time-to-live) instead of explicit deletes
```

## When to Use Cassandra

### Best Fit For

- **Massive scale**: Terabytes to petabytes of data
- **High write throughput**: Millions of writes per second
- **Geo-distribution**: Multi-data center, global distribution
- **Always-on applications**: Zero downtime requirements
- **Time-series**: IoT, metrics, events
- **Wide-column data**: Variable attributes per row

### Not Ideal For

- **Complex queries**: Ad-hoc analytics, complex joins
- **ACID transactions**: Cassandra is eventually consistent by default
- **Small scale**: Use simpler databases when scale isn't needed
- **Document storage**: Consider MongoDB instead

### Real-World Use Cases

| Company | Use Case |
|---------|----------|
| Netflix | Read/Write cache for metadata, recommendations |
| Apple | iCloud, messages, app store search |
| Discord | Messages, user data |
| Instagram | Activity logs, social graph |
| Uber | Trip history, events |

## Common Mistakes

### 1. Not Designing for Queries

```sql
-- BAD: Relational thinking
CREATE TABLE entities (
    id UUID PRIMARY KEY,
    type TEXT,
    data JSON
);
-- Now how do you efficiently query?

-- GOOD: Query-first design
CREATE TABLE entities_by_type (
    type TEXT,
    id UUID,
    data JSON,
    PRIMARY KEY (type, id)
);
```

### 2. Hot Partitions

```sql
-- BAD: All data in one partition
PRIMARY KEY (constant_value, id);

-- GOOD: Distribute across partitions
PRIMARY KEY (user_id, id);
```

### 3. Ignoring Data Modeling Upfront

```sql
-- Don't just migrate SQL schema
-- Design from scratch based on queries
```

### 4. Using High Cardinality Partition Keys Inefficiently

```sql
-- Each user gets their own partition (OK if moderate writes)
PRIMARY KEY (user_id)

-- Each event gets partition (bad if millions per user!)
PRIMARY KEY (event_id)
```

## Summary

Cassandra is designed for massive scale:

- **Masterless architecture** provides no single point of failure
- **Query-first modeling** requires different thinking than relational
- **Tunable consistency** allows balancing availability and correctness
- **Denormalization** is expected, not avoided
- **Partition design** is critical for performance
