# Cassandra Overview

## Overview

Apache Cassandra is a distributed, decentralized, linearly scalable NoSQL database designed to handle large amounts of data across multiple commodity servers with no single point of failure.

---

## Problem Statement

Traditional databases struggle with massive scale and high availability requirements. Cassandra addresses these challenges with its distributed architecture, making it ideal for time-series data, IoT applications, and systems requiring continuous uptime.

---

## Main Content Section 1

### Key Concepts

* **Distributed**: Data replicated across multiple nodes and data centers
* **Masterless**: No single point of failure
* **Tunable Consistency**: Balance between consistency and performance
* **Wide-Column Store**: Flexible schema with row-based partitioning
* **CQL**: Cassandra Query Language (similar to SQL)

---

## Main Content Section 2

### Architecture Components

* **Partition Key**: Determines data distribution across nodes
* **Cluster**: Collection of nodes (data centers)
* **Replication Factor**: Number of copies of each row
* **Consistency Level**: How many nodes must respond for read/write
* **Gossip Protocol**: Node communication for cluster state

## Code Example

```sql
-- Create keyspace with replication
CREATE KEYSPACE myapp
WITH REPLICATION = {
  'class' : 'NetworkTopologyStrategy',
  'datacenter1' : 3
};

-- Create table with primary key
CREATE TABLE users (
  user_id UUID PRIMARY KEY,
  email TEXT,
  name TEXT,
  created_at TIMESTAMP
);

-- Create table with composite partition key
CREATE TABLE orders (
  customer_id UUID,
  order_id TIMEUUID,
  product TEXT,
  amount DECIMAL,
  PRIMARY KEY (customer_id, order_id)
) WITH CLUSTERING ORDER BY (order_id DESC);

-- Query with partition key (efficient)
SELECT * FROM orders WHERE customer_id = ?;
```

## Best Practices

* Model around query patterns (queries first, then schema)
* Use partition keys wisely to avoid hot spots
* Denormalize for read performance
* Use lightweight transactions sparingly
* Plan for data growth and partitioning

---

## Common Mistakes

* Using high-cardinality partition keys
* Creating too many indexes (secondary indexes)
* Allowing unbounded partitions
* Ignoring tombstones in deleted data
* Underestimating the importance of data modeling upfront

---

## Summary

Cassandra excels at handling massive scale and high availability requirements. Success requires understanding its distributed nature and designing data models around query patterns rather than relationships.
