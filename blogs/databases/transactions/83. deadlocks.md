# Database Deadlocks

## Overview

A deadlock occurs when two or more transactions are waiting for each other to release locks. Neither can proceed, causing the database to detect and resolve the situation, typically by rolling back one transaction.

---

## Problem Statement

Deadlocks can halt critical operations, especially under high concurrency. While databases have built-in deadlock handling, applications should minimize the risk and handle deadlocks gracefully.

---

## Main Content Section 1

### How Deadlocks Occur

* Transaction A holds lock on resource 1, needs lock on resource 2
* Transaction B holds lock on resource 2, needs lock on resource 1
* Neither can proceed until the other releases its lock
* Database detects circular wait and breaks it

---

## Main Content Section 2

### Detection vs Prevention

* **Detection**: DB periodically checks for deadlocks, resolves by rolling back victim
* **Prevention**: DB uses lock ordering or timestamps to prevent deadlocks

## Code Example

```sql
-- Deadlock example: Two transactions competing for resources
-- Transaction 1:
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;  -- Lock row 1
UPDATE accounts SET balance = balance + 100 WHERE id = 2;  -- Needs row 2

-- Transaction 2 (concurrent):
BEGIN;
UPDATE accounts SET balance = balance - 50 WHERE id = 2;  -- Lock row 2
UPDATE accounts SET balance = balance + 50 WHERE id = 1;  -- Needs row 1, DEADLOCK!

-- Database picks victim and rolls back:
-- ERROR: could not obtain lock on row in relation "accounts"

-- Application handling with retry:
try:
    while retries < 3:
        try:
            execute_transaction()
            break
        except DeadlockError:
            retries += 1
            sleep(random.uniform(0.1, 0.5))
except MaxRetriesExceeded:
    raise "Transaction failed after max retries"
```

---

## Best Practices

* Always acquire locks in consistent order across transactions
* Keep transactions short to reduce lock hold time
* Use lower isolation levels when possible
* Access tables in the same order in all transactions
* Implement retry logic for deadlock handling

---

## Common Mistakes

* Not ordering lock acquisition consistently
* Long-running transactions holding many locks
* Ignoring deadlock errors in application code
* Overusing explicit locking (SELECT FOR UPDATE)
* Not monitoring deadlock frequency

---

## Summary

Deadlocks are inevitable in concurrent systems but manageable. Consistent lock ordering, short transactions, and proper error handling keep systems running smoothly.
