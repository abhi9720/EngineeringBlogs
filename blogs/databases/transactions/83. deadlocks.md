# Database Deadlocks: Understanding, Preventing, and Handling

## Overview

A **deadlock** in database systems occurs when two or more transactions are waiting for each other to release locks, creating a circular dependency where none of them can proceed. This is one of the most challenging concurrency problems to deal with because it can bring critical operations to a halt, especially under high concurrency.

While databases have built-in mechanisms to detect and resolve deadlocks (typically by rolling back one of the transactions), frequent deadlocks indicate fundamental issues in your application design that should be addressed. Understanding deadlocks deeply is essential for building robust, high-concurrency database applications.

Deadlocks differ from simple lock contention: in lock contention, transactions queue up waiting for a lock, but progress is possible once the lock is released. In a deadlock, there's no progress at all without database intervention.

---

## How Deadlocks Occur

### The Classic Scenario

A deadlock typically follows a four-step pattern:

1. **Transaction A** acquires a lock on Resource 1
2. **Transaction B** acquires a lock on Resource 2
3. **Transaction A** now needs to acquire a lock on Resource 2 (but it's held by B)
4. **Transaction B** now needs to acquire a lock on Resource 1 (but it's held by A)

Neither transaction can proceed because each is waiting for a resource held by the other. This is a **circular wait**—the defining characteristic of a deadlock.

### Real-World Example: Money Transfer

```sql
-- Transaction 1: Transfer from Account A to Account B
BEGIN;
UPDATE accounts SET balance = balance - 500 WHERE id = 1;  -- Locks row A
-- Now needs to lock row B to complete transfer

-- Transaction 2 (concurrent): Transfer from Account B to Account A
BEGIN;
UPDATE accounts SET balance = balance - 300 WHERE id = 2;  -- Locks row B
-- Now needs to lock row A to complete transfer

-- DEADLOCK: Each transaction holds a lock the other needs
```

At this point, neither transaction can proceed. The database's deadlock detector must intervene.

### More Complex Scenarios

Deadlocks can involve more than two transactions:

```
Transaction A: Holds lock on Resource 1, needs Resource 2
Transaction B: Holds lock on Resource 2, needs Resource 3
Transaction C: Holds lock on Resource 3, needs Resource 1

-- Three-way deadlock
```

Deadlocks can also involve multiple lock types:
- **Row locks** vs. **table locks**
- **Shared locks** vs. **exclusive locks**
- **Intent locks** (metadata locks in some databases)

---

## Types of Deadlocks

### 1. Transaction Deadlocks

The most common type—transactions waiting on each other to release row or table locks. These are what we typically mean when discussing database deadlocks.

### 2. Latch Deadlocks

Internal database structures (latches) used to protect in-memory data structures can deadlock. These are often harder to diagnose because they occur at a lower level than application-visible locks.

### 3. Metadata Locks

DDL operations (ALTER TABLE, CREATE INDEX) can deadlock with concurrent DML or other DDL operations. These locks often have different semantics than regular row locks.

### 4. Distributed Deadlocks

In distributed systems with multiple database nodes, deadlocks can span across servers. Detecting these requires coordination between nodes and is more complex.

## Deadlock Detection vs. Prevention

### Deadlock Detection

Most modern databases use **deadlock detection**—periodically scanning the wait-for graph to identify cycles (circular waits).

**How it works**:
1. The database maintains a **wait-for graph** showing which transactions are waiting for which locks
2. A background daemon periodically checks for cycles in this graph
3. When a cycle is found, the database selects a **victim transaction** to roll back
4. The victim is rolled back, releasing its locks and allowing others to proceed

**Victim Selection**: When choosing which transaction to roll back, databases typically consider:
- Transaction age (older transactions may have invested more work)
- Lock count (transaction holding fewer locks is often a better choice)
- User-defined priorities (some databases support this)
- SQL Server: Uses a cost-based approach
- MySQL InnoDB: Rolls back the transaction that completed the least amount of work
- PostgreSQL: Similar approach, considers transaction age and lock count

```sql
-- PostgreSQL: View current locks and waiting transactions
SELECT blocked_locks.pid     AS blocked_pid,
       blocked_activity.usename  AS blocked_user,
       blocking_locks.pid     AS blocking_pid,
       blocking_activity.usename AS blocking_user,
       blocked_activity.query    AS blocked_statement,
       blocking_activity.query   AS blocking_statement
FROM pg_catalog.pg_locks         AS blocked_locks
JOIN pg_catalog.pg_stat_activity AS blocked_activity ON blocked_locks.pid = blocked_activity.pid
JOIN pg_catalog.pg_locks         AS blocking_locks ON blocking_locks.locktype = blocked_locks.locktype
JOIN pg_catalog.pg_stat_activity AS blocking_activity ON blocking_locks.pid = blocking_activity.pid
WHERE NOT blocked_locks.granted;
```

### Deadlock Prevention

Some databases implement **deadlock prevention** strategies that avoid deadlocks entirely by ensuring one transaction backs out before a deadlock can form.

**Wait-Die Prevention**: Older transactions wait for younger ones; younger transactions die (rollback) immediately.

**Wound-Wait Prevention**: Older transactions "wound" (rollback) younger transactions that request a lock they hold; younger transactions wait for older ones.

These strategies rely on transaction timestamps and can prevent deadlocks but may cause more rollbacks than detection-based approaches.

**Implementation**:
- **Oracle**: Uses a variation of wait-die/wound-wait called "statement-level deadlock prevention"
- **SQL Server**: Primarily uses detection
- **PostgreSQL**: Uses detection with timeout as backup

## Database-Specific Handling

### PostgreSQL

PostgreSQL uses deadlock detection with a configurable timeout:
- **deadlock_timeout**: Default 1s—how long to wait before checking for deadlocks
- Automatic victim selection based on transaction age and lock count

```sql
-- Configure deadlock detection
SET deadlock_timeout = '2s';

-- View deadlock information in logs
-- PostgreSQL logs: "detected deadlock on lock..." with details

-- Query to find blocking queries
SELECT * FROM pg_stat_activity 
WHERE state = 'active' 
AND wait_event_type = 'Lock';
```

### MySQL (InnoDB)

MySQL InnoDB has sophisticated deadlock detection:
- **innodb_lock_wait_timeout**: How long to wait for a lock (default 50s)
- Automatic detection and victim rollback
- Dumps deadlock information to error log

```sql
-- Configure
SET GLOBAL innodb_lock_wait_timeout = 10;

-- View InnoDB status to see recent deadlocks
SHOW ENGINE INNODB STATUS;

-- The output includes:
-- LATEST DETECTED DEADLOCK section with full transaction details
```

### SQL Server

SQL Server uses detection with:
- **Lock Timeout**: Can be set per-query
- **Deadlock Priority**: SET DEADLOCK_PRIORITY to control which transaction rolls back
- ** deadlock_timeout** configuration

```sql
-- Set lock timeout
SET LOCK_TIMEOUT 5000;  -- 5 seconds

-- Set deadlock priority
SET DEADLOCK_PRIORITY LOW;  -- This connection rolls back first

-- Trace flag 1222 for detailed deadlock information
DBCC TRACEON (1222);
```

## Strategies to Prevent Deadlocks

### 1. Consistent Lock Ordering

**The most important strategy**: Always acquire locks in a consistent order across all transactions.

```sql
-- BAD: Different order in different transactions
-- Transaction 1: A then B
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;  -- Lock A
UPDATE accounts SET balance = balance + 100 WHERE id = 2;  -- Lock B (blocked)
COMMIT;

-- Transaction 2: B then A (different order!)
BEGIN;
UPDATE accounts SET balance = balance - 50 WHERE id = 2;  -- Lock B
UPDATE accounts SET balance = balance + 50 WHERE id = 1;  -- Lock A (DEADLOCK)
COMMIT;

-- GOOD: Consistent order (always A then B)
-- Transaction 1: A then B
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;  -- Lock A
UPDATE accounts SET balance = balance + 100 WHERE id = 2;  -- Lock B (OK)
COMMIT;

-- Transaction 2: A then B
BEGIN;
UPDATE accounts SET balance = balance - 50 WHERE id = 1;  -- Lock A (waits, but no deadlock!)
UPDATE accounts SET balance = balance + 50 WHERE id = 2;  -- Lock B (after A releases)
COMMIT;
```

### 2. Keep Transactions Short

Shorter transactions hold locks for less time, reducing deadlock probability.

**Best practices**:
- Do pre-validation before starting the transaction
- Process in batches if needed
- Avoid user interaction during transactions
- Use optimistic concurrency control when appropriate

```sql
-- BAD: Long transaction with human interaction
BEGIN;
-- User thinks for 10 minutes...
UPDATE orders SET status = 'processed' WHERE id = 123;
COMMIT;

-- GOOD: Validate before transaction
SELECT balance INTO v_balance FROM accounts WHERE id = 1;
IF v_balance < required_amount THEN
    RAISE EXCEPTION 'Insufficient funds';
END IF;

BEGIN;
-- Quick update
UPDATE accounts SET balance = balance - amount WHERE id = 1;
COMMIT;
```

### 3. Access Tables in Consistent Order

When accessing multiple tables, use the same order everywhere:

```sql
-- BAD: Different table order in different places
-- Function A
INSERT INTO orders (...) VALUES (...);
INSERT INTO order_items (...) VALUES (...);

-- Function B (called in different context)
INSERT INTO order_items (...) VALUES (...);
INSERT INTO orders (...) VALUES (...);

-- GOOD: Always orders then order_items
INSERT INTO orders (...) VALUES (...);
INSERT INTO order_items (...) VALUES (...);
```

### 4. Use Lower Isolation Levels When Possible

Higher isolation levels (like Serializable) use more locks and increase deadlock probability. Use the lowest level that meets your correctness requirements.

### 5. Use Optimistic Concurrency Control

For low-contention scenarios, use version-based updates instead of pessimistic locking:

```sql
-- Using version for optimistic concurrency
UPDATE accounts 
SET balance = balance - @amount, 
    version = version + 1 
WHERE id = @account_id 
AND version = @expected_version;

IF ROW_COUNT() = 0 THEN
    -- Handle conflict: another transaction modified the row
    -- Retry or show conflict message
END IF;
```

### 6. Avoid Cursors in Transactions

Cursor operations can hold locks for extended periods. Use set-based operations instead.

## Handling Deadlocks in Applications

### Retry Logic

Your application should be prepared to handle deadlock errors gracefully with automatic retries:

```python
import random
import time
from contextlib import retrying

@retrying(retry_on_exception=Exception, stop_max_attempt_number=3, wait_exponential_multiplier=1000)
def execute_with_retry():
    try:
        with connection.cursor() as cursor:
            cursor.execute("""
                BEGIN;
                UPDATE accounts SET balance = balance - %s WHERE id = %s;
                UPDATE accounts SET balance = balance + %s WHERE id = %s;
                COMMIT;
            """, (amount, from_id, amount, to_id))
    except Exception as e:
        # Check if it's a deadlock
        if is_deadlock_error(e):
            raise  # Let retrying handle it
        raise  # Re-raise other errors

def is_deadlock_error(e):
    # Database-specific error codes
    deadlock_codes = {
        '40001',  # SQL State for deadlock
        '1213',   # MySQL deadlock
        '5701',   # PostgreSQL deadlock
    }
    return str(e).startswith(tuple(deadlock_codes))
```

```java
// Java example with retry
public void transferFunds(String fromAccount, String toAccount, BigDecimal amount) {
    int maxRetries = 3;
    int retryCount = 0;
    
    while (retryCount < maxRetries) {
        try {
            performTransfer(fromAccount, toAccount, amount);
            return;
        } catch (SQLException e) {
            if (isDeadlock(e) && retryCount < maxRetries - 1) {
                retryCount++;
                // Exponential backoff
                Thread.sleep((long) Math.pow(2, retryCount) * 100);
            } else {
                throw e;
            }
        }
    }
}
```

### Exponential Backoff

When retrying after deadlocks, use exponential backoff to avoid thundering herd:

```python
import random

def execute_with_backoff(max_retries=3):
    for attempt in range(max_retries):
        try:
            return do_operation()
        except DeadlockError:
            if attempt == max_retries - 1:
                raise
            # Exponential backoff with jitter
            sleep_time = (2 ** attempt) * random.uniform(0.1, 0.5)
            time.sleep(sleep_time)
```

## Monitoring and Debugging

### Key Metrics to Track

1. **Deadlock frequency**: How often deadlocks occur
2. **Deadlock rate**: Deadlocks per second/minute
3. **Average deadlock resolution time**: How long deadlocks block operations
4. **Victim transaction duration**: How much work is lost when a transaction is rolled back

### PostgreSQL Monitoring

```sql
-- Check for recent deadlocks
SELECT pid, now() - xact_start AS duration, query, state
FROM pg_stat_activity
WHERE state != 'idle'
AND xact_start IS NOT NULL
ORDER BY duration DESC;

-- Configure logging
SET log_lock_waits = on;
SET deadlock_timeout = '1s';
-- Check pg_log for deadlock messages
```

### MySQL Monitoring

```sql
-- Show engine status (includes deadlock info)
SHOW ENGINE INNODB STATUS;

-- Enable error logging
-- Check error log for: "InnoDB: Deadlock found when trying to get lock"

-- Information schema
SELECT * FROM information_schema.INNODB_TRX;
```

### SQL Server Monitoring

```sql
-- Extended events for deadlock capture
CREATE EVENT SESSION deadlock_monitor ON SERVER
ADD EVENT sqlserver.xml_deadlock_report
ADD TARGET package0.event_file(SET filename='deadlocks.xel');

-- Query deadlock information from system tables
SELECT * FROM sys.dm_exec_requests WHERE blocking_session_id > 0;

-- Trace flags
DBCC TRACEON (1222, 3605);  -- Detailed deadlock logging
```

## Common Mistakes and Pitfalls

### 1. Not Having Retry Logic

**Problem**: Application crashes when deadlock occurs instead of retrying.

**Solution**: Implement automatic retry with exponential backoff.

### 2. Inconsistent Lock Ordering

**Problem**: Different code paths acquire locks in different orders, causing deadlocks.

**Solution**: Enforce consistent ordering at the application architecture level.

### 3. Long-Running Transactions

**Problem**: Transactions hold locks for extended periods, increasing deadlock probability.

**Solution**: Keep transactions short; validate before starting the transaction.

### 4. Ignoring Deadlocks

**Problem**: Deadlocks are treated as rare events and not investigated.

**Solution**: Monitor deadlock metrics; investigate root causes when deadlocks occur.

### 5. Over-Indexing

**Problem**: Too many indexes cause more lock acquisitions and deadlocks.

**Solution**: Review and remove unused indexes; consider lock implications.

### 6. Mixing DDL with DML

**Problem**: ALTER TABLE during peak traffic causes metadata locks and deadlocks.

**Solution**: Schedule DDL during maintenance windows; use online schema changes when available.

## Best Practices Summary

1. **Design for no deadlocks**: Use consistent lock ordering from the start
2. **Keep transactions short**: Every second counts in reducing deadlock probability
3. **Implement retry logic**: Handle deadlock errors gracefully
4. **Monitor proactively**: Track deadlock metrics and investigate when they increase
5. **Use appropriate isolation**: Don't use higher isolation than needed
6. **Test under load**: Simulate concurrent access to find deadlock-prone code
7. **Log for debugging**: Capture deadlock details for post-mortem analysis
8. **Document the behavior**: Ensure team members understand how to handle deadlocks

---

## Summary

Deadlocks are an inevitable part of concurrent database systems, but they are manageable with proper design and handling. The key insights are:

- **Prevention is better than cure**: Consistent lock ordering, short transactions, and appropriate isolation levels prevent most deadlocks
- **Detection handles the rest**: Databases detect and resolve remaining deadlocks by rolling back victims
- **Applications must handle failures**: Implement retry logic to handle rolled-back transactions gracefully
- **Monitoring is essential**: Track deadlock metrics to identify and fix patterns

By understanding how deadlocks occur and implementing these strategies, you can build robust database applications that handle concurrent access gracefully.
