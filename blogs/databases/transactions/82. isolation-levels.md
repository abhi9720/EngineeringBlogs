# Isolation Levels

## Overview

Transaction isolation levels define how concurrent transactions interact. Higher isolation reduces anomalies but can decrease performance due to locking. Choosing the right level balances consistency needs with throughput.

---

## Problem Statement

Concurrent transactions can cause various anomalies: dirty reads, non-repeatable reads, phantom reads. Isolation levels provide configurable protection against these while managing locking overhead.

---

## Main Content Section 1

### Isolation Levels (Low to High)

* **Read Uncommitted**: Lowest isolation, allows dirty reads
* **Read Committed**: Only reads committed data (default in most DBs)
* **Repeatable Read**: Same read returns same result within transaction
* **Serializable**: Executes as if transactions ran sequentially

---

## Main Content Section 2

### Anomalies Addressed

* **Dirty Read**: Reading uncommitted data from another transaction
* **Non-Repeatable Read**: Re-reading data shows different values
* **Phantom Read**: New rows appear in re-executed range query
* **Lost Update**: Concurrent updates overwrite each other

---

## Code Example

```sql
-- Set isolation level for session
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- PostgreSQL isolation level options
BEGIN;
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;

-- Example showing non-repeatable read
-- Session 1:
BEGIN;
SELECT balance FROM accounts WHERE id = 1;  -- Returns 1000

-- Session 2:
BEGIN;
UPDATE accounts SET balance = 2000 WHERE id = 1;
COMMIT;

-- Session 1:
SELECT balance FROM accounts WHERE id = 1;  -- With Read Committed: returns 2000
                                            -- With Repeatable Read: returns 1000
COMMIT;

-- Example of phantom read
-- Session 1:
BEGIN;
SELECT COUNT(*) FROM orders WHERE date > '2024-01-01';  -- Returns 100

-- Session 2:
INSERT INTO orders VALUES (...);  -- New row added

-- Session 1:
SELECT COUNT(*) FROM orders WHERE date > '2024-01-01';  -- Returns 101
```

---

## Best Practices

* Use lowest isolation level that meets your requirements
* Understand the specific anomalies each level prevents
* Monitor lock contention at higher isolation levels
* Consider optimistic concurrency for read-heavy workloads
* Document the chosen isolation level per transaction type

---

## Common Mistakes

* Using Serializable without understanding performance cost
* Ignoring isolation level defaults (often Read Committed)
* Not testing with concurrent transactions
* Assuming Read Committed is sufficient for all use cases
* Over-using SERIALIZABLE when optimistic locking suffices

---

## Summary

Isolation levels trade off between consistency and performance. Understanding each level's guarantees and anomalies helps choose the right one for your application's requirements.
