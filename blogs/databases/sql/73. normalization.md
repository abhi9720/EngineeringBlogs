# Database Normalization: A Complete Guide

## Overview

Database normalization is the systematic process of organizing data in a relational database to reduce redundancy, eliminate data anomalies, and ensure data integrity. The process involves structuring tables and defining relationships according to increasingly stringent rules called **normal forms**.

Understanding normalization is fundamental to good database design. While the concepts can seem abstract, they directly address real-world problems that cause data corruption, inconsistent reports, and maintenance nightmares.

---

## Why Normalization Matters

### The Problems Without Normalization

When data is not properly normalized, three types of anomalies occur:

**1. Update Anomaly**: When you need to change information, you might need to update multiple rows.

```sql
-- Imagine a table with customer address stored in every order
-- If customer moves, you need to update every order row!
UPDATE orders SET customer_address = '123 New St' 
WHERE customer_id = 123;
-- Must update ALL rows where this customer has orders!
```

**2. Insert Anomaly**: You cannot insert certain data without also inserting other unrelated data.

```sql
-- Can't add a new customer without placing an order
-- Can't add a product without having a category
```

**3. Delete Anomaly**: Deleting one piece of data unintentionally removes other related data.

```sql
-- Deleting last order from a customer removes customer information too
-- Deleting a product category might delete all products in that category
```

### The Goals of Normalization

- **Eliminate redundant data**: Store each piece of information in only one place
- **Ensure data dependencies**: Data is stored in the right tables, related to the right keys
- **Improve data integrity**: Constraints enforce relationships and valid values
- **Simplify maintenance**: Updates only need to happen in one place

## The Normal Forms

### First Normal Form (1NF)

**Definition**: A table is in 1NF when:
1. Each column contains atomic (indivisible) values
2. Each column contains values of a single type
3. Each row is unique
4. There are no repeating groups or arrays

**What's not allowed**:
- Multiple values in a single column (e.g., "123 Main St, Apt 4")
- Storing lists or arrays in columns
- Repeating columns like phone1, phone2, phone3

```sql
-- NOT IN 1NF: Storing multiple phone numbers in one column
CREATE TABLE customers (
    id INT PRIMARY KEY,
    name VARCHAR(100),
    phones VARCHAR(200)  -- Contains "555-1234,555-5678,555-9012"
);

-- IN 1NF: Separate phone numbers into child table
CREATE TABLE customers (
    id INT PRIMARY KEY,
    name VARCHAR(100)
);

CREATE TABLE customer_phones (
    customer_id INT,
    phone_number VARCHAR(20),
    PRIMARY KEY (customer_id, phone_number)
);
```

**Another example**:
```sql
-- NOT IN 1NF: Repeating columns
CREATE TABLE order_items_bad (
    order_id INT,
    item1_id INT,
    item1_qty INT,
    item2_id INT,
    item2_qty INT,
    item3_id INT,
    item3_qty INT  -- What if there are 4 items?
);

-- IN 1NF: One row per item
CREATE TABLE order_items (
    order_id INT,
    item_id INT,
    quantity INT,
    PRIMARY KEY (order_id, item_id)
);
```

---

### Second Normal Form (2NF)

**Definition**: A table is in 2NF when:
1. It is in 1NF
2. No non-key column is dependent on only part of a composite key (for tables with composite keys)

**This only applies to tables with composite primary keys**. Tables with single-column primary keys that are in 1NF are automatically in 2NF.

```sql
-- NOT IN 2NF: Partial dependency on composite key
-- Primary key: (order_id, product_id)
CREATE TABLE order_items_bad (
    order_id INT,
    product_id INT,
    product_name VARCHAR(100),  -- Depends ONLY on product_id, not on order_id!
    quantity INT,
    price DECIMAL(10,2),
    PRIMARY KEY (order_id, product_id)
);

-- In 2NF: Separate product information
CREATE TABLE order_items (
    order_id INT,
    product_id INT,
    quantity INT,
    price DECIMAL(10,2),
    PRIMARY KEY (order_id, product_id)
);

CREATE TABLE products (
    id INT PRIMARY KEY,
    name VARCHAR(100)
);
```

**Transitive dependency explanation**:
- If (A, B) is the primary key
- And A → C (A determines C)
- Then C is partially dependent on the key

---

### Third Normal Form (3NF)

**Definition**: A table is in 3NF when:
1. It is in 2NF
2. No non-key column is transitively dependent on the primary key
3. All non-key columns depend directly on the primary key, not on other non-key columns

**Transitive dependency**: When A → B and B → C, then A → C (transitively).

```sql
-- NOT IN 3NF: Transitive dependency
-- Primary key: student_id
CREATE TABLE students_bad (
    student_id INT PRIMARY KEY,
    name VARCHAR(100),
    course_id INT,
    course_name VARCHAR(100),  -- Depends on course_id, not directly on student_id!
    grade VARCHAR(2)
);

-- In 3NF: Separate courses
CREATE TABLE students (
    student_id INT PRIMARY KEY,
    name VARCHAR(100),
    course_id INT
);

CREATE TABLE courses (
    course_id INT PRIMARY KEY,
    course_name VARCHAR(100)
);

CREATE TABLE enrollments (
    student_id INT,
    course_id INT,
    grade VARCHAR(2),
    PRIMARY KEY (student_id, course_id)
);
```

---

### Boyce-Codd Normal Form (BCNF)

**Definition**: A table is in BCNF when:
1. It is in 3NF
2. Every determinant (column that determines other columns) must be a candidate key

This is a stricter version of 3NF that handles edge cases where 3NF still allows anomalies.

```sql
-- In 3NF but not BCNF
-- Table: (student, course, instructor)
-- Primary key: (student, course)
-- But: instructor determines course (every course has one instructor)

-- If this is the key:
-- - student + course uniquely identifies a row
-- - instructor determines course
-- This violates BCNF because instructor is not a candidate key

CREATE TABLE enrollments_bad (
    student VARCHAR(50),
    course VARCHAR(50),
    instructor VARCHAR(50),
    grade VARCHAR(2),
    PRIMARY KEY (student, course)
);

-- In BCNF: Separate instructor information
CREATE TABLE courses (
    course VARCHAR(50) PRIMARY KEY,
    instructor VARCHAR(50)
);

CREATE TABLE enrollments (
    student VARCHAR(50),
    course VARCHAR(50),
    grade VARCHAR(2),
    PRIMARY KEY (student, course),
    FOREIGN KEY (course) REFERENCES courses(course)
);
```

## Achieving Normal Forms

### Step-by-Step Process

```
1. Start with unnormalized data
      ↓
2. Identify and remove repeating groups → 1NF
      ↓
3. Identify and remove partial dependencies → 2NF
      ↓
4. Identify and remove transitive dependencies → 3NF
      ↓
5. Check for BCNF violations → BCNF (if needed)
```

### Practical Example: Online Store

**Starting point (unnormalized)**:

```sql
-- Each row contains: order info, customer info, product info
CREATE TABLE orders_unnormalized (
    order_id INT,
    order_date DATE,
    customer_name VARCHAR(100),
    customer_email VARCHAR(100),
    customer_city VARCHAR(50),
    product_name VARCHAR(100),
    product_category VARCHAR(50),
    quantity INT,
    price DECIMAL(10,2)
);
-- Problem: Multiple rows per order, duplicated customer info, duplicated product info
```

**After 1NF** (remove repeating groups, atomic values):

```sql
CREATE TABLE orders (
    order_id INT PRIMARY KEY,
    order_date DATE,
    customer_name VARCHAR(100),
    customer_email VARCHAR(100),
    customer_city VARCHAR(50)
);

CREATE TABLE order_items (
    order_id INT,
    product_name VARCHAR(100),
    product_category VARCHAR(50),
    quantity INT,
    price DECIMAL(10,2),
    PRIMARY KEY (order_id, product_name)
);
```

**After 2NF** (remove partial dependencies):

```sql
CREATE TABLE orders (
    order_id INT PRIMARY KEY,
    order_date DATE,
    customer_id INT  -- Reference to customers table
);

CREATE TABLE customers (
    id INT PRIMARY KEY,
    name VARCHAR(100),
    email VARCHAR(100),
    city VARCHAR(50)
);

CREATE TABLE products (
    id INT PRIMARY KEY,
    name VARCHAR(100),
    category_id INT
);

CREATE TABLE order_items (
    order_id INT,
    product_id INT,
    quantity INT,
    price DECIMAL(10,2),
    PRIMARY KEY (order_id, product_id)
);
```

**After 3NF** (remove transitive dependencies):

```sql
-- Customer information is now independent of orders
CREATE TABLE customers (
    id INT PRIMARY KEY,
    name VARCHAR(100),
    email VARCHAR(100),
    city_id INT  -- Reference to cities
);

CREATE TABLE cities (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    state VARCHAR(50)
);

-- Product category is now independent
CREATE TABLE products (
    id INT PRIMARY KEY,
    name VARCHAR(100),
    category_id INT
);

CREATE TABLE categories (
    id INT PRIMARY KEY,
    name VARCHAR(50)
);
```

## When to Denormalize

Normalization isn't always the answer. Denormalization (intentionally adding redundancy) can improve read performance in specific scenarios.

### Common Denormalization Patterns

**1. Storing Calculated Values**

```sql
-- Normalized: Calculate total on the fly
SELECT SUM(quantity * price) FROM order_items WHERE order_id = 123;

-- Denormalized: Store calculated total in orders table
ALTER TABLE orders ADD COLUMN total DECIMAL(10,2);

-- When order is placed or modified, update this column
-- Trade-off: Must maintain consistency (trigger or application code)
```

**2. Adding Redundant Columns for Joins**

```sql
-- Normalized: Join to get customer name
SELECT o.*, c.name
FROM orders o
INNER JOIN customers c ON o.customer_id = c.id;

-- Denormalized: Store customer_name in orders
ALTER TABLE orders ADD COLUMN customer_name VARCHAR(100);

-- Trade-off: Must update when customer name changes
```

**3. Flattening Hierarchies**

```sql
-- Normalized: Parent-child relationship
CREATE TABLE categories (
    id INT PRIMARY KEY,
    parent_id INT REFERENCES categories(id),
    name VARCHAR(50)
);

-- Denormalized: Store full path
ALTER TABLE categories ADD COLUMN path VARCHAR(500);  -- "/electronics/computers/laptops"

-- Trade-off: Must update path when hierarchy changes
```

### When Denormalization Makes Sense

- **Read-heavy workloads**: Where complex joins are too slow
- **Reporting databases**: Where query simplicity matters more than write performance
- **Caching at database level**: Materialized pre-computed results
- **Historical data**: Where normalization rules can be relaxed

**Rule of thumb**: Start normalized, denormalize only when you have measured performance issues that normalization causes.

## Practical Guidelines

### What Normal Form to Aim For?

| Application Type | Recommended Normal Form |
|-----------------|------------------------|
| Most OLTP applications | 3NF |
| Read-heavy reporting | 3NF with selective denormalization |
| Data warehousing | Often less normalized (2NF or 1NF) |
| Simple applications | 2NF may be sufficient |

### Decision Framework

1. **Start with 3NF**: Good balance for most applications
2. **Check for BCNF**: If you have overlapping candidate keys
3. **Denormalize only after measuring**: Not preemptively
4. **Document your schema decisions**: Why you normalized or denormalized

## Common Mistakes

### 1. Over-Normalizing

Creating too many tables leads to excessive joins and complex queries.

```sql
-- Too normalized: Splitting too finely
-- Customer has separate tables for: name, address, phone, email
-- Join heaven becomes a nightmare!
```

**Solution**: Balance normalization with query patterns.

### 2. Premature Denormalization

Adding redundancy before measuring actual performance problems.

**Solution**: Measure first, then optimize.

### 3. Ignoring Foreign Keys

Not enforcing relationships properly.

```sql
-- Bad: No foreign key constraint
CREATE TABLE orders_bad (
    customer_id INT  -- No constraint linking to customers
);

-- Good: Proper foreign key
CREATE TABLE orders (
    customer_id INT REFERENCES customers(id)
);
```

### 4. Not Considering the Full Query Pattern

Normalizing for write performance but making reads slow.

**Solution**: Consider both read and write patterns.

---

## Summary

Database normalization is a fundamental skill for database design:

- **1NF**: Atomic values, no repeating groups
- **2NF**: No partial dependencies on composite keys
- **3NF**: No transitive dependencies
- **BCNF**: Every determinant is a candidate key

Most applications should aim for 3NF as a baseline, with BCNF for critical data integrity. Denormalization should be a measured decision based on actual performance requirements, not a default approach.
