# Database Normalization

## Overview

Database normalization is the process of organizing data to reduce redundancy and improve data integrity. It involves structuring tables and defining relationships according to specific rules called normal forms.

---

## Problem Statement

Poorly structured databases lead to data redundancy, update anomalies, insertion problems, and deletion issues. Normalization provides a systematic approach to organizing data efficiently.

---

## Main Content Section 1

### Normal Forms

* **1NF (First Normal Form)**: Eliminate repeating groups, atomic values
* **2NF (Second Normal Form)**: Remove partial dependencies on composite keys
* **3NF (Third Normal Form)**: Remove transitive dependencies
* **BCNF (Boyce-Codd Normal Form)**: Handle overlapping candidate keys

---

## Main Content Section 2

### Normalization Example

```
Unnormalized: Order contains customer name, address, multiple items
1NF: Separate order items into a child table
2NF: Ensure non-key attributes depend on entire key
3NF: Remove attributes that depend on non-key attributes
```

---

## Code Example

```sql
-- 1NF: Atomic values, no repeating groups
-- Bad: phone_numbers VARCHAR containing "123,456,789"
-- Good: Separate phone_numbers table or array column

-- 2NF: No partial dependencies
-- For composite key (order_id, item_id), item_name only depends on item_id

-- 3NF: No transitive dependencies
-- Bad: student_id -> grade -> gpa_points
-- Good: student_id -> grade, and grade -> gpa_points as separate tables

-- BCNF: Every determinant must be a candidate key
-- For table (student, course, instructor), if instructor determines course
-- and (student, course) is the only key, this violates BCNF
```

---

## Best Practices

* Start with 3NF as a baseline for most applications
* Consider BCNF for critical data integrity requirements
* Document your normalization decisions
* Balance normalization with performance needs

---

## Common Mistakes

* Over-normalizing leading to excessive joins
* Ignoring denormalization for read-heavy scenarios
* Missing foreign key constraints
* Forgetting that real-world data often requires compromises

---

## Summary

Normalization is essential for data integrity but should be balanced with practical performance considerations. Know when to apply higher normal forms and when strategic denormalization is appropriate.
