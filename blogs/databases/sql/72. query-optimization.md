# Database Query Optimization: The Complete Guide

## Overview

Query optimization is the art and science of making SQL queries run faster. A well-optimized query can be orders of magnitude faster than a poorly written one—transforming a query that takes seconds into one that takes milliseconds. Understanding how databases execute queries and how to influence that execution is essential for building high-performance applications.

Database systems use a **query optimizer** to determine the most efficient way to execute a query. The optimizer considers many factors: available indexes, table statistics, join order, and more. However, the optimizer's choices are only as good as the information it has and the query structure you provide.

---

## How Query Optimizers Work

### The Optimizer's Job

When you submit a SQL query, the database doesn't just execute it as written. Instead, it goes through several phases:

1. **Parsing**: Convert SQL into an internal representation (parse tree)
2. **Semantic Analysis**: Check that tables and columns exist, permissions are valid
3. **Query Rewrite**: Apply rules to simplify and transform the query
4. **Cost-Based Planning**: Generate execution plans and choose the cheapest
5. **Execution**: Run the chosen plan

### Cost-Based Optimization

Modern optimizers use **cost-based planning**. Each possible execution plan is assigned a cost estimate based on:
- Estimated number of rows to process
- I/O cost (disk reads)
- CPU cost (comparison operations)
- Memory usage
- Network cost (for distributed databases)

The optimizer chooses the plan with the lowest estimated cost.

```sql
-- The optimizer might choose between:
-- Plan A: Full table scan → filter → sort
-- Plan B: Index scan → index-ordered retrieval
-- Plan C: Materialize subquery → join

-- EXPLAIN shows which plan the optimizer chose
EXPLAIN SELECT * FROM orders WHERE customer_id = 123;
```

### Statistics Are Critical

The optimizer uses **table statistics** to make decisions:
- Row count
- Value distribution (histograms)
- Index availability and selectivity
- Data distribution

**Always keep statistics up to date**:

```sql
-- PostgreSQL
ANALYZE orders;

-- MySQL
ANALYZE TABLE orders;

-- SQL Server
UPDATE STATISTICS orders;
```

## Analyzing Query Execution

### Using EXPLAIN

The `EXPLAIN` command shows the execution plan without actually running the query. `EXPLAIN ANALYZE` actually executes the query and shows real runtime statistics.

**PostgreSQL**:
```sql
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT o.*, c.name
FROM orders o
INNER JOIN customers c ON o.customer_id = c.id
WHERE o.order_date >= '2024-01-01'
ORDER BY o.total DESC
LIMIT 10;
```

**Key metrics to look for**:
- **Seq Scan** vs **Index Scan**: Sequential scans are usually slower for large tables
- **Nested Loop** vs **Hash Join** vs **Merge Join**: Different algorithms suit different data sizes
- **Sort**: In-memory vs disk-based sorting
- **Actual vs Estimated rows**: Large discrepancies indicate stale statistics

**MySQL**:
```sql
EXPLAIN FORMAT=JSON
SELECT * FROM orders WHERE customer_id = 123;
```

### Understanding Execution Plan Elements

**Scan Types**:
- **Seq Scan**: Read entire table, one row at a time
- **Index Scan**: Use index to find rows, then fetch
- **Index Only Scan**: Satisfy query entirely from index (no table access)
- **Bitmap Scan**: Use bitmap index combinations

**Join Types**:
- **Nested Loop**: For each row in outer table, scan inner table
- **Hash Join**: Build hash table from smaller table, probe with larger
- **Merge Join**: Both inputs sorted, merge in single pass

**Sort Operations**:
- **Quicksort**: In-memory sort for small datasets
- **External Sort**: Disk-based sort for large datasets (very slow)
- **Index Order**: Using a sorted index avoids sorting entirely

```sql
-- If you see "Seq Scan on orders" for a query that filters by indexed column,
-- it might mean:
-- 1. No index exists on that column
-- 2. The index is not selective enough (table scan is faster)
-- 3. Statistics are outdated
```

## Optimization Techniques

### 1. Select Only Needed Columns

```sql
-- BAD: Fetches all columns
SELECT * FROM orders WHERE id = 123;

-- GOOD: Fetches only needed columns
SELECT id, order_date, total, status FROM orders WHERE id = 123;
```

**Why it matters**:
- Reduces data transfer
- Can enable index-only scans
- Less memory usage
- Better for covering indexes

### 2. Use WHERE Clauses Effectively

**Filter Early**: Apply WHERE conditions as soon as possible to reduce intermediate result sizes.

```sql
-- GOOD: Filter first, then join
SELECT *
FROM (
    SELECT * FROM orders WHERE order_date >= '2024-01-01'
) o
INNER JOIN customers c ON o.customer_id = c.id;

-- This is often faster because it reduces rows before the join
```

**Avoid Functions on Indexed Columns**:

```sql
-- BAD: Function prevents index usage
SELECT * FROM orders
WHERE YEAR(order_date) = 2024;

-- GOOD: Range condition uses index
SELECT * FROM orders
WHERE order_date >= '2024-01-01' AND order_date < '2025-01-01';
```

**Avoid Leading Wildcards in LIKE**:

```sql
-- BAD: Leading wildcard can't use index
SELECT * FROM products WHERE name LIKE '%widget%';

-- GOOD: Can use index if available
SELECT * FROM products WHERE name LIKE 'widget%';

-- For full text search, use full-text indexes instead
```

### 3. Use Appropriate Joins

**Use JOIN Instead of Subqueries When Appropriate**:

```sql
-- Subquery approach (sometimes slower)
SELECT name
FROM customers
WHERE id IN (
    SELECT customer_id FROM orders WHERE total > 1000
);

-- JOIN approach (often faster)
SELECT DISTINCT c.name
FROM customers c
INNER JOIN orders o ON c.id = o.customer_id
WHERE o.total > 1000;
```

**But Understand When Subqueries Are Better**:
- When you need aggregation from one table to filter another
- When the subquery is very selective
- When using EXISTS or NOT EXISTS

### 4. Index Design for Queries

**Create Indexes Based on Query Patterns**:

```sql
-- If this is a common query:
SELECT * FROM orders WHERE customer_id = ? AND status = 'pending';

-- Create a composite index:
CREATE INDEX idx_orders_customer_status 
ON orders(customer_id, status);

-- Column order matters: put equality conditions first
-- If you often query just by customer_id:
-- Make customer_id first in the index
```

**Use Covering Indexes**:

```sql
-- If you frequently run:
SELECT id, order_date, total FROM orders WHERE customer_id = 123;

-- Create covering index:
CREATE INDEX idx_orders_customer_covering 
ON orders(customer_id) INCLUDE (order_date, total);
```

### 5. Pagination Optimization

**Offset-Based Pagination Has Problems**:

```sql
-- Gets slower as offset increases
SELECT * FROM orders ORDER BY id LIMIT 10 OFFSET 10000;
-- Has to scan 10,010 rows to return 10!
```

**Cursor-Based Pagination Is Better**:

```sql
-- Instead of OFFSET, use keyset pagination
-- First page:
SELECT * FROM orders ORDER BY id LIMIT 10;

-- Subsequent pages using last seen ID:
SELECT * FROM orders 
WHERE id > 10000
ORDER BY id 
LIMIT 10;
-- Much faster! Uses index on id
```

### 6. Avoid OR in WHERE Clauses

```sql
-- Can sometimes prevent index usage
SELECT * FROM orders WHERE status = 'pending' OR total > 1000;

-- Use UNION instead:
SELECT * FROM orders WHERE status = 'pending'
UNION
SELECT * FROM orders WHERE total > 1000;
-- Each part can use its own index
```

### 7. Use EXISTS Instead of IN for Subqueries

```sql
-- IN can be inefficient with large subqueries
SELECT * FROM customers WHERE id IN (SELECT customer_id FROM orders);

-- EXISTS often optimizes better
SELECT * FROM customers c WHERE EXISTS (
    SELECT 1 FROM orders WHERE customer_id = c.id
);
```

## Common Performance Anti-Patterns

### 1. SELECT *

```sql
-- BAD: Returns all columns
SELECT * FROM orders WHERE id = 123;

-- GOOD: Returns only needed columns
SELECT id, order_date, total FROM orders WHERE id = 123;
```

### 2. No Filtering in Subqueries

```sql
-- BAD: Correlated subquery runs for each row
SELECT *,
    (SELECT COUNT(*) FROM order_items WHERE order_id = orders.id) AS item_count
FROM orders;

-- GOOD: Join or derived table
SELECT o.*, oi.item_count
FROM orders o
LEFT JOIN (
    SELECT order_id, COUNT(*) AS item_count
    FROM order_items
    GROUP BY order_id
) oi ON o.id = oi.order_id;
```

### 3. Not Using Proper Data Types

```sql
-- Storing dates as strings prevents date operations and index usage
-- BAD:
ALTER TABLE orders ADD COLUMN order_date_string VARCHAR(20);

-- GOOD:
ALTER TABLE orders ADD COLUMN order_date TIMESTAMP;
```

### 4. Implicit Type Conversion

```sql
-- If customer_id is numeric but compared to string:
SELECT * FROM orders WHERE customer_id = '123';
-- Database must convert string to number for each row!

-- Use proper type:
SELECT * FROM orders WHERE customer_id = 123;
```

### 5. Nested Views

```sql
-- Views layered on views can prevent optimization
CREATE VIEW v1 AS SELECT * FROM orders WHERE total > 100;
CREATE VIEW v2 AS SELECT * FROM v1 WHERE status = 'pending';
CREATE VIEW v3 AS SELECT * FROM v2 WHERE order_date > '2024-01-01';

-- Query on view v3 may not push down filters efficiently
-- Consider materialized views or direct queries instead
```

## Database-Specific Optimization

### PostgreSQL

```sql
-- Enable query timing
\timing on

-- Parallel query for large tables
SET max_parallel_workers_per_gather = 4;

-- Force specific index usage
SELECT * FROM orders SET (enable_seqscan = off);

-- Create useful extensions
CREATE EXTENSION pg_stat_statements;
-- Monitor slow queries
SELECT query, calls, mean_time 
FROM pg_stat_statements 
ORDER BY mean_time DESC 
LIMIT 10;
```

### MySQL

```sql
-- Force index usage
SELECT * FROM orders FORCE INDEX (idx_customer);

-- Optimize table (rebuild and analyze)
OPTIMIZE TABLE orders;

-- Check query cache (MySQL 5.7 and earlier)
SHOW VARIABLES LIKE 'query_cache_%';
```

### SQL Server

```sql
-- Show actual execution plan (Ctrl+M in SSMS)

-- Create plan guide for specific query patterns
EXEC sp_create_plan_guide 
    @name = N'Guide_Orders_Select',
    @stmt = N'SELECT * FROM orders WHERE customer_id = @cid',
    @type = N'SQL',
    @module_or_batch = NULL,
    @params = NULL,
    @hints = N'OPTION (OPTIMIZE FOR (@cid = 12345))';

-- Index rebuild
ALTER INDEX ALL ON orders REBUILD;
```

## Monitoring and Finding Slow Queries

### PostgreSQL

```sql
-- Enable slow query logging
ALTER SYSTEM SET log_min_duration_statement = 1000;  -- Log queries > 1s

-- pg_stat_statements
SELECT * FROM pg_stat_statements 
WHERE query LIKE '%orders%'
ORDER BY total_time DESC;

-- Find missing indexes
SELECT 
    schemaname,
    relname,
    seq_scan,
    idx_scan
FROM pg_stat_user_tables
WHERE seq_scan > idx_scan
ORDER BY seq_scan DESC;
```

### MySQL

```sql
-- Enable slow query log
SET GLOBAL slow_query_log = 'ON';
SET GLOBAL long_query_time = 1;

-- View slow queries
SELECT * FROM mysql.slow_log;

-- Show process list
SHOW PROCESSLIST;
-- Kill long-running query
KILL 123;
```

### SQL Server

```sql
-- Query Store (SQL Server 2016+)
ALTER DATABASE database_name SET QUERY_STORE = ON;

-- Find expensive queries
SELECT 
    qsq.query_text,
    rs.*
FROM sys.query_store_query qsq
JOIN sys.query_store_runtime_stats rs ON qsq.query_id = rs.query_id
ORDER BY rs.avg_duration DESC;

-- Missing index suggestions
SELECT * FROM sys.dm_db_missing_index_details;
```

## Summary

Query optimization is both a science and an art. Key principles:

- **Measure first**: Use EXPLAIN to understand what's happening
- **Index strategically**: Create indexes for your actual query patterns
- **Filter early**: Reduce data as soon as possible in the query
- **Select only what you need**: Avoid SELECT *
- **Keep statistics updated**: Analytics depend on accurate data
- **Monitor continuously**: Track query performance over time
- **Test changes**: Measure before and after optimizations
