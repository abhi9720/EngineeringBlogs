# Database Indexing: A Comprehensive Guide to Query Performance

## Overview

Database indexes are data structures that dramatically improve the speed of data retrieval operations. Think of an index as similar to a book's index—it allows you to quickly find information without scanning every page. In database terms, indexes provide quick lookup paths to data rows, transforming queries from O(n) full table scans into O(log n) or even O(1) operations.

However, indexes come with a cost: they require additional storage space and must be maintained whenever data changes. Understanding how to create and use indexes effectively is one of the most impactful skills for database performance optimization.

---

## How Indexes Work

### The Basics

At its core, an index stores a sorted copy of the indexed column values along with pointers to the actual data rows. This sorted structure enables efficient binary search operations that dramatically reduce the number of rows that need to be examined.

**Without an index**: The database must read every row in the table to find matching records (full table scan).

**With an index**: The database navigates the index structure to quickly locate matching rows, reading only a small fraction of the total data.

### B-Tree Indexes: The Workhorse

B-Tree (Balanced Tree) indexes are the most common type and serve as the default for most database systems. Their balanced nature ensures consistent performance regardless of where data is located.

**Structure**: B-Tree indexes maintain data in a balanced tree structure with multiple levels:
- **Root**: The top level, pointing to child nodes
- **Branch nodes**: Intermediate levels containing index values and pointers
- **Leaf nodes**: The bottom level containing the actual index values and row pointers

**Search complexity**: O(log n) where n is the number of rows—extremely efficient even for millions of rows.

```sql
-- Visual representation of B-Tree structure
-- For an index on (last_name, first_name):

                    [M]
                   /   \
             [G-L]       [R-Z]
            /    \       /    \
        [A-C]  [D-F]  [M-O]  [P-S]  [T-Z]
         |      |      |      |      |
        ...   ...    ...    ...    ...
```

**Operations supported**:
- Equality comparisons (=, IN)
- Range comparisons (<, >, BETWEEN)
- Sorting (ORDER BY) when index matches
- Prefix searches (LIKE 'abc%')

### Hash Indexes

Hash indexes use a hash function to map index values to bucket locations. They provide O(1) lookup for exact match queries but cannot support range queries.

**Best for**: Equality lookups on high-cardinality columns

**Limitations**:
- No range queries
- No ordering support
- Cannot use for LIKE prefix searches

```sql
-- Creating a hash index (PostgreSQL example)
CREATE INDEX idx_user_email_hash ON users USING HASH (email);
```

### When Indexes Are Used

The database query optimizer decides whether to use an index based on:
1. **Statistics**: Distribution of values in the column
2. **Query patterns**: What columns are referenced in WHERE, JOIN, ORDER BY
3. **Table size**: Whether using an index is faster than a table scan
4. **Cost estimation**: Comparing estimated costs of different plans

## Types of Indexes

### Single-Column Indexes

Index on one column. The most straightforward type.

```sql
CREATE INDEX idx_customers_email ON customers(email);
```

**Best for**: Columns frequently used alone in WHERE clauses.

### Composite (Multi-Column) Indexes

Index on two or more columns. The order of columns matters significantly.

```sql
CREATE INDEX idx_orders_customer_date ON orders(customer_id, order_date);
```

**Column order matters**: The index can support queries that use:
- All indexed columns
- The first column only
- First and second columns (in that order)

But NOT queries that use only the second column or third column alone.

```sql
-- Index on (customer_id, order_date, status)

-- Can use index efficiently for:
WHERE customer_id = 123
WHERE customer_id = 123 AND order_date > '2024-01-01'
WHERE customer_id = 123 AND order_date = '2024-01-01' AND status = 'shipped'

-- Cannot use index efficiently for:
WHERE order_date = '2024-01-01'  -- First column not used
WHERE status = 'shipped'         -- First column not used
```

**Design guidelines**:
- Put high-cardinality columns first (for filtering)
- Put equality conditions first, range conditions last
- Consider your query patterns when ordering

### Unique Indexes

Ensures all values in the indexed column(s) are distinct. Automatically created for primary key constraints.

```sql
CREATE UNIQUE INDEX idx_users_email ON users(email);
```

### Partial (Filtered) Indexes

Index only a subset of rows based on a condition. Saves space and improves performance for specific queries.

```sql
-- PostgreSQL: Index only active customers
CREATE INDEX idx_active_customers ON customers(email) WHERE status = 'active';

-- Useful for queries that always filter on the same condition
SELECT * FROM customers WHERE status = 'active' AND email LIKE '%@example.com';
```

### Covering Indexes

An index that includes all columns needed by a query, allowing the database to satisfy the query entirely from the index without accessing the table (index-only scan).

```sql
-- If you frequently query: SELECT id, email, name FROM users WHERE email = ?
CREATE INDEX idx_users_email_covering ON users(email, id, name);

-- Query can be satisfied entirely from the index
SELECT id, email, name FROM users WHERE email = 'john@example.com';
```

### Full-Text Indexes

For efficient text search across large text content.

```sql
-- MySQL
ALTER TABLE articles ADD FULLTEXT(title, content);
SELECT * FROM articles WHERE MATCH(title, content) AGAINST('database optimization');

-- PostgreSQL
CREATE INDEX idx_articles_fts ON articles USING GIN(to_tsvector('english', title || ' ' || content));
```

### Spatial Indexes

For geographic and geometric data types.

```sql
-- PostgreSQL with PostGIS
CREATE INDEX idx_locations_geo ON locations USING GIST (location);

-- MySQL
CREATE SPATIAL INDEX idx_locations_geo ON locations(location);
```

## Index Selectivity and Cardinality

### Understanding Selectivity

**Selectivity** is the ratio of distinct values to total rows. Higher selectivity means fewer rows match any given value, making indexes more effective.

**High cardinality**: Many distinct values (e.g., unique IDs, email addresses)
- Very selective, indexes are very effective

**Low cardinality**: Few distinct values (e.g., boolean flags, status codes)
- Less selective, indexes may not be beneficial

```sql
-- Check cardinality
SELECT column_name, COUNT(DISTINCT column_name) AS distinct_values,
       (SELECT COUNT(*) FROM table_name) AS total_rows
FROM table_name
GROUP BY column_name;
```

**When indexes help most**:
- High cardinality columns in WHERE clauses
- Columns used in JOIN conditions
- Columns used in ORDER BY

**When indexes help less**:
- Low cardinality columns (e.g., gender, boolean flags)
- Columns rarely used in queries
- Tables that are mostly read via full scans

### Index Usage Analysis

```sql
-- PostgreSQL: Check if index is being used
EXPLAIN ANALYZE
SELECT * FROM orders WHERE customer_id = 123;

-- Look for "Index Scan using idx_orders_customer" in the output

-- MySQL: Check query execution
EXPLAIN
SELECT * FROM orders WHERE customer_id = 123;

-- Look for "type: ref" or "type: range" instead of "type: ALL" (full scan)
```

## Index Maintenance

### The Write Overhead

Every INSERT, UPDATE, and DELETE must maintain all indexes on the table. This means:
- More indexes = slower writes
- Each index adds overhead proportional to data changes

```sql
-- Creating 5 indexes vs 1 index on the same table
INSERT INTO products (name, category, price, description, stock) 
VALUES ('Widget', 'Electronics', 19.99, 'A great widget', 100);

-- With 1 index: 1 write + 1 index update = 2 operations
-- With 5 indexes: 1 write + 5 index updates = 6 operations
```

### When to Add Indexes

**Add indexes for**:
- Columns frequently used in WHERE clauses
- Columns used in JOIN conditions (foreign keys)
- Columns used in ORDER BY
- High-cardinality columns

**Avoid indexes for**:
- Columns rarely queried
- Low-cardinality columns (unless combined with other conditions)
- Tables with heavy write loads and rare reads
- Large blob/text columns

### Monitoring Index Usage

```sql
-- PostgreSQL: Find unused indexes
SELECT
    indexrelname,
    idx_scan,
    idx_tup_read,
    idx_tup_fetch
FROM pg_stat_user_indexes
WHERE indexrelname NOT LIKE '%pkey%'
ORDER BY idx_scan ASC;

-- If idx_scan is 0 or very low, the index might be unused
```

## Database-Specific Index Types

### PostgreSQL

- **B-Tree**: Default, handles equality and range
- **GiST**: Generalized Search Tree—spatial, full-text
- **GIN**: Generalized Inverted Index—arrays, JSONB, full-text
- **BRIN**: Block Range Index—very large datasets with sequential data
- **Hash**: For equality comparisons

```sql
-- GiST for spatial data
CREATE INDEX idx_locations ON locations USING GIST (location);

-- BRIN for time-series data
CREATE INDEX idx_events_time ON events USING BRIN (created_at);

-- GIN for JSONB
CREATE INDEX idx_documents_data ON documents USING GIN (data);
```

### MySQL (InnoDB)

- **B-Tree**: Primary index type
- **Full-text**: For text search
- **Spatial**: For geographic data

**Auto-incrementing primary keys** create a "right-growing" B-Tree that is highly efficient for inserts.

### SQL Server

- **Clustered**: Determines physical storage order (one per table)
- **Non-clustered**: Additional indexes (multiple per table)
- **Columnstore**: For data warehouse analytics
- **Full-text**: For text search

```sql
-- Create clustered index (determines table storage order)
CREATE CLUSTERED INDEX idx_orders_id ON orders(order_id);

-- Create non-clustered indexes
CREATE NONCLUSTERED INDEX idx_orders_customer ON orders(customer_id);
CREATE NONCLUSTERED INDEX idx_orders_date ON orders(order_date);
```

### MongoDB

- **Single field**: Index on one field
- **Compound**: Multiple fields (order matters)
- **Multi-key**: Arrays (automatic)
- **Text**: Full-text search
- **2dsphere**: Geospatial
- **2d**: Legacy geospatial

```javascript
// MongoDB indexes
db.collection.createIndex({ email: 1 });                    // Single field
db.collection.createIndex({ customerId: 1, date: -1 });    // Compound
db.collection.createIndex({ content: "text" });            // Full-text
```

## Best Practices

### 1. Index Based on Query Patterns

Analyze your application's most frequent and critical queries:

```sql
-- If this is a common query:
SELECT * FROM orders WHERE status = 'pending' AND customer_id = 123;

-- Create an index that supports it:
CREATE INDEX idx_orders_status_customer ON orders(status, customer_id);
-- Or if customer_id is used more:
CREATE INDEX idx_orders_customer_status ON orders(customer_id, status);
```

### 2. Consider Composite Index Column Order

Put equality conditions first, range conditions last:

```sql
-- Query: WHERE status = 'active' AND created_at > '2024-01-01'
-- Good index: (status, created_at)
-- Bad index: (created_at, status) - range on first column is inefficient

-- Query: WHERE customer_id = 123 AND status IN ('pending', 'processing')
-- Good index: (customer_id, status)
```

### 3. Use Covering Indexes for Frequent Queries

```sql
-- If you frequently run:
SELECT id, email, name FROM users WHERE email = ?;

-- Create covering index:
CREATE INDEX idx_users_email_covering ON users(email) INCLUDE (id, name);
-- Or in some databases:
CREATE INDEX idx_users_email_covering ON users(email, id, name);
```

### 4. Monitor and Remove Unused Indexes

```sql
-- Regular maintenance: Check for unused indexes
-- Remove indexes with 0 or very few scans
DROP INDEX idx_unused_index;
```

### 5. Use Partial Indexes for Specific Queries

```sql
-- If you often query active users:
SELECT * FROM users WHERE status = 'active' AND email LIKE '%@company.com';

-- Partial index:
CREATE INDEX idx_active_users_email ON users(email) WHERE status = 'active';
```

### 6. Consider Index on Foreign Keys

Always index foreign key columns to improve join performance:

```sql
-- If orders.customer_id references customers.id
CREATE INDEX idx_orders_customer ON orders(customer_id);
```

### 7. Balance Read and Write Performance

- More indexes = faster reads = slower writes
- Remove indexes that aren't being used
- Add indexes strategically as you identify performance issues

## Common Mistakes

### 1. Indexing Every Column

**Problem**: Creating indexes on all columns slows down writes and wastes space.

**Solution**: Only index columns that are actually used in queries.

### 2. Ignoring Index Column Order

**Problem**: Wrong column order in composite indexes makes them ineffective.

**Solution**: Put high-selectivity columns first, equality before range.

### 3. Not Considering Query Patterns

**Problem**: Creating indexes without understanding how queries actually run.

**Solution**: Analyze query patterns using logs and EXPLAIN before creating indexes.

### 4. Using Low-Cardinality Columns

**Problem**: Indexing columns like "status" or "is_active" provides minimal benefit.

**Solution**: Avoid indexing low-cardinality columns unless combined with other conditions.

### 5. Not Updating Statistics

**Problem**: Outdated statistics lead to poor index usage decisions.

**Solution**: Regularly update table statistics (ANALYZE, UPDATE STATISTICS).

### 6. Assuming Indexes Are Always Good

**Problem**: Blindly adding indexes without measuring impact.

**Solution**: Benchmark before and after, monitor for regressions.

## Code Examples

### Common Index Creation Patterns

```sql
-- Basic index
CREATE INDEX idx_products_category ON products(category);

-- Composite index for specific query pattern
CREATE INDEX idx_orders_customer_date_status 
ON orders(customer_id, order_date, status);

-- Unique index
CREATE UNIQUE INDEX idx_users_email ON users(email);

-- Partial index
CREATE INDEX idx_active_orders ON orders(customer_id) 
WHERE status = 'active';

-- Covering index for specific query
CREATE INDEX idx_customer_orders_covering 
ON orders(customer_id, order_date, total) 
INCLUDE (status, created_at);

-- Index with custom sort order (descending)
CREATE INDEX idx_products_price_desc ON products(price DESC);
```

### Analyzing Index Usage

```sql
-- PostgreSQL: Find slow queries and check index usage
EXPLAIN (ANALYZE, BUFFERS)
SELECT o.*, c.name 
FROM orders o 
JOIN customers c ON o.customer_id = c.id 
WHERE o.order_date >= '2024-01-01';

-- Look for:
-- "Index Scan" or "Index Only Scan" = good
-- "Seq Scan" on large table = might need index

-- MySQL: Check query plan
EXPLAIN FORMAT=JSON
SELECT * FROM orders 
WHERE customer_id = 123 AND status = 'shipped';
```

## Summary

Indexes are essential tools for database performance optimization, but they must be used thoughtfully. Key takeaways:

- **B-Tree indexes** are the most versatile and commonly used
- **Composite index column order** significantly impacts effectiveness
- **Selectivity matters**—high-cardinality columns benefit most
- **Every index has a cost**—maintenance overhead on writes
- **Monitor and maintain**—remove unused indexes, add needed ones
- **Use covering indexes** when possible for index-only scans
