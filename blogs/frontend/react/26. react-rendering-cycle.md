# React Rendering Cycle: Complete Guide

## Overview

React's rendering cycle is the heart of the library. Understanding how React decides when and how to update the DOM is essential for building performant applications. This guide covers the complete rendering flow, reconciliation, and optimization strategies.

---

## How React Rendering Works Internally

### The Rendering Flow

```
┌─────────────────────────────────────────────────────────────────┐
│                    React Rendering Flow                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. STATE CHANGE                                               │
│     │                                                          │
│     ▼                                                          │
│  2. RENDER PHASE (Virtual DOM)                                 │
│     │  React calls component functions                         │
│     │  Builds new virtual DOM tree                             │
│     │  Compares with previous (reconciliation)               │
│     │                                                          │
│     ▼                                                          │
│  3. COMMIT PHASE (Real DOM)                                    │
│     │  Apply only the necessary changes                        │
│     │  Update actual DOM nodes                                 │
│     │  Run useLayoutEffect, ref callbacks                     │
│     │                                                          │
│     ▼                                                          │
│  4. CYCLE COMPLETE                                             │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### Virtual DOM

```javascript
// React creates a JavaScript representation of the DOM
// Virtual DOM is a tree of React elements (plain objects)

const element = (
    <div className="container">
        <h1>Hello</h1>
        <p>World</p>
    </div>
);

// This creates this virtual DOM:
{
    type: 'div',
    props: {
        className: 'container',
        children: [
            { type: 'h1', props: { children: 'Hello' } },
            { type: 'p', props: { children: 'World' } }
        ]
    }
}
```

### Reconciliation Algorithm

```javascript
// React compares old and new virtual DOM trees
// Key heuristics:
// 1. Different types = different components (rebuild)
// 2. Same type, different keys = replace
// 3. Same type, same key = update

// Example: When props change, component re-renders
function Counter() {
    const [count, setCount] = useState(0);

    // This triggers re-render of Counter
    // Children don't automatically re-render unless affected
    return <div>{count}</div>;
}

// Key insight: Parent re-renders trigger child re-renders by default
function Parent() {
    const [count, setCount] = useState(0);

    return (
        <div onClick={() => setCount(count + 1)}>
            <Child />  // Re-renders even though props don't change!
        </div>
    );
}
```

## Real-World React Use Cases

### Case 1: useMemo for Expensive Calculations

```javascript
// Problem: Expensive calculation runs on every render
function ExpensiveComponent({ items, filter }) {
    // This runs on EVERY render, even if items/filter haven't changed!
    const filtered = items
        .filter(item => item.name.includes(filter))
        .sort((a, b) => a.price - b.price);

    return (
        <ul>
            {filtered.map(item => <li key={item.id}>{item.name}</li>)}
        </ul>
    );
}

// Solution: useMemo - memoizes expensive calculations
function OptimizedComponent({ items, filter }) {
    const filtered = useMemo(() => {
        return items
            .filter(item => item.name.includes(filter))
            .sort((a, b) => a.price - b.price);
    }, [items, filter]);  // Recalculate only when deps change

    return (
        <ul>
            {filtered.map(item => <li key={item.id}>{item.name}</li>)}
        </ul>
    );
}
```

### Case 2: useCallback for Stable Function References

```javascript
// Problem: New function created on every render, causing child re-renders
function Parent() {
    const [count, setCount] = useState(0);

    const handleClick = () => {  // New function every render!
        console.log('clicked', count);
    };

    return <Child onClick={handleClick} count={count} />;
}

function Child({ onClick, count }) {
    // Child re-renders whenever parent renders (props changed!)
    return <button onClick={onClick}>Count: {count}</button>;
}

// Solution: useCallback - stable function reference
function OptimizedParent() {
    const [count, setCount] = useState(0);

    const handleClick = useCallback(() => {
        console.log('clicked', count);
    }, [count]);  // Only new function when count changes

    return <Child onClick={handleClick} count={count} />;
}

// Better: Pass stable reference to memoized component
const ChildMemo = React.memo(Child);
```

### Case 3: React.memo for Component Memoization

```javascript
// Problem: Child re-renders even when props haven't changed
function Parent({ name }) {
    const [count, setCount] = useState(0);
    return (
        <div>
            <h1>{name}</h1>
            <button onClick={() => setCount(c => c + 1)}>Increment</button>
            <Child name={name} />
        </div>
    );
}

function Child({ name }) {
    console.log('Child rendered');  // Logs every time parent renders!
    return <p>Hello {name}</p>;
}

// Solution: React.memo - skip re-render if props are same
const ChildMemo = React.memo(function Child({ name }) {
    console.log('Child rendered');  // Only when name actually changes
    return <p>Hello {name}</p>;
});

// With custom comparison
const ChildWithComparison = React.memo(function Child({ data }) {
    return <div>{data.value}</div>;
}, (prevProps, nextProps) => {
    // Return true to skip re-render
    return prevProps.data.id === nextProps.data.id;
});
```

### Case 4: useReducer for Complex State Logic

```javascript
// Problem: Multiple related state updates
function BadCounter() {
    const [count, setCount] = useState(0);
    const [step, setStep] = useState(1);

    const increment = () => setCount(c => c + step);
    const decrement = () => setCount(c => c - step);
    const reset = () => { setCount(0); setStep(1); };

    return (
        <div>
            <button onClick={increment}>+</button>
            <span>{count}</span>
            <button onClick={decrement}>-</button>
            <button onClick={reset}>Reset</button>
        </div>
    );
}

// Solution: useReducer - grouped state updates
const initialState = { count: 0, step: 1 };

function reducer(state, action) {
    switch (action.type) {
        case 'increment':
            return { ...state, count: state.count + state.step };
        case 'decrement':
            return { ...state, count: state.count - state.step };
        case 'reset':
            return { ...state, count: 0, step: 1 };
        case 'setStep':
            return { ...state, step: action.payload };
        default:
            return state;
    }
}

function GoodCounter() {
    const [state, dispatch] = useReducer(reducer, initialState);

    return (
        <div>
            <button onClick={() => dispatch({ type: 'increment' })}>+</button>
            <span>{state.count}</span>
            <button onClick={() => dispatch({ type: 'decrement' })}>-</button>
            <button onClick={() => dispatch({ type: 'reset' })}>Reset</button>
        </div>
    );
}
```

## Production Considerations

### 1. Component Composition

```javascript
// Avoid prop drilling with composition
function App() {
    return (
        <ThemeProvider>
            <AuthProvider>
                <Router>
                    <Layout>
                        <Routes>
                            <Route path="/" element={<HomePage />} />
                        </Routes>
                    </Layout>
                </Router>
            </AuthProvider>
        </ThemeProvider>
    );
}

// Use Context for shared state, but be careful with re-renders
const ThemeContext = React.createContext();

function ThemeProvider({ children }) {
    const [theme, setTheme] = useState('light');

    return (
        <ThemeContext.Provider value={{ theme, setTheme }}>
            {children}
        </ThemeContext.Provider>
    );
}

// Consuming components only re-render when context value changes
function ThemedButton() {
    const { theme, setTheme } = useContext(ThemeContext);

    return (
        <button onClick={() => setTheme(theme === 'light' ? 'dark' : 'light')}>
            Current: {theme}
        </button>
    );
}
```

## Common Mistakes

### Mistake 1: Not Understanding When Re-renders Occur

```javascript
// WRONG: Thinking only prop changes cause re-renders
function Parent() {
    const [count, setCount] = useState(0);

    // Passing object - new reference every render!
    const user = { name: 'John', age: 30 };

    return <Child user={user} />;
}

function Child({ user }) {
    console.log('Rendered');  // Every click!
    return <div>{user.name}</div>;
}

// CORRECT: Pass primitive values or memoize
function ParentFixed() {
    const [count, setCount] = useState(0);
    const name = 'John';  // Primitive - stable reference

    return <Child name={name} count={count} />;
}
```

### Mistake 2: Forgetting Dependencies in Hooks

```javascript
// WRONG: Missing dependency in useEffect
function BrokenComponent() {
    const [data, setData] = useState(null);

    useEffect(() => {
        fetchData().then(result => setData(result));
        // Missing dependency - may use stale closure!
    }, []);  // Empty deps but uses fetchData

    return <div>{data?.value}</div>;
}

// CORRECT: Include all dependencies
function FixedComponent() {
    const [data, setData] = useState(null);

    useEffect(() => {
        let cancelled = false;

        fetchData().then(result => {
            if (!cancelled) setData(result);
        });

        return () => { cancelled = true; };
    }, [fetchData]);  // Dependency included!

    return <div>{data?.value}</div>;
}
```

### Mistake 3: Using Index as Key

```javascript
// WRONG: Using index as key causes bugs with reordering
function List({ items, onMoveUp }) {
    return (
        <ul>
            {items.map((item, index) => (
                <li key={index}>
                    {item.name}
                    <button onClick={() => onMoveUp(index)}>↑</button>
                </li>
            ))}
        </ul>
    );
}

// When moving item up, React thinks items changed because indices changed!

// CORRECT: Use stable unique IDs
function FixedList({ items, onMoveUp }) {
    return (
        <ul>
            {items.map(item => (
                <li key={item.id}>
                    {item.name}
                    <button onClick={() => onMoveUp(item.id)}>↑</button>
                </li>
            ))}
        </ul>
    );
}
```

### Mistake 4: Mutating State Directly

```javascript
// WRONG: Mutating state doesn't trigger re-render
function Broken() {
    const [items, setItems] = useState([{ id: 1, name: 'A' }]);

    const addItem = () => {
        items.push({ id: Date.now(), name: 'New' });
        // setItems(items); // Even with setItems, mutating doesn't work well
    };

    return <button onClick={addItem}>Add</button>;
}

// CORRECT: Create new array
function Fixed() {
    const [items, setItems] = useState([{ id: 1, name: 'A' }]);

    const addItem = () => {
        setItems([...items, { id: Date.now(), name: 'New' }]);
        // Or: setItems(prev => [...prev, newItem])
    };

    return <button onClick={addItem}>Add</button>;
}
```

## Summary

1. **Render Phase**: React calls components, builds virtual DOM
2. **Reconciliation**: Compares old and new virtual DOM
3. **Commit Phase**: Applies changes to real DOM
4. **Memoization**: useMemo, useCallback, React.memo prevent unnecessary renders

Key takeaways:
- Use React.memo for components that re-render with same props
- Use useMemo for expensive calculations
- Use useCallback for function references passed to memoized components
- Avoid index as key - use stable IDs

Understanding the rendering cycle helps you write performant React applications.
