# JavaScript Closures: The Complete Guide

## Overview

A closure is a function that has access to variables from its outer (enclosing) scope, even after the outer function has returned. Closures are fundamental to JavaScript and are the foundation for many patterns: data privacy, function factories, currying, and maintaining state in async code.

---

## How Closures Work Internally

### Lexical Environment

Every JavaScript function creates a closure. When a function is defined, it captures the environment in which it was created—the lexical environment:

```javascript
function outer() {
    const outerVar = 'I am from outer';

    function inner() {
        console.log(outerVar);  // inner has access to outerVar!
    }

    return inner;
}

const closure = outer();  // outer returns, but closure still works
closure();  // "I am from outer"
```

### The Scope Chain

When JavaScript looks up a variable, it searches the scope chain:

```
┌─────────────────────────────────────────────────────────────┐
│                    Execution Context                        │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  closure's scope                                            │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  inner's scope                                       │   │
│  │  ┌───────────────────────────────────────────────┐  │   │
│  │  │                                              │  │   │
│  │  │  Does not find 'outerVar' here              │  │   │
│  │  └───────────────────────────────────────────────┘  │   │
│  │         │                                            │   │
│  │         │ Looks up to...                             │   │
│  └─────────┼────────────────────────────────────────────┘   │
│            │                                              │
│  ┌─────────┴────────────────────────────────────────────┐  │
│  │  outer's scope (lexical environment)                  │  │
│  │                                                      │  │
│  │  outerVar = 'I am from outer'  ← FOUND!             │  │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### Closure Memory

```javascript
function createCounter() {
    let count = 0;  // Private variable

    return {
        increment() {
            count++;
            return count;
        },
        decrement() {
            count--;
            return count;
        },
        getCount() {
            return count;
        }
    };
}

const counter = createCounter();
console.log(counter.increment());  // 1
console.log(counter.increment());  // 2
console.log(counter.getCount());   // 2
// count is encapsulated - not accessible directly!
```

## Real-World JavaScript Use Cases

### Case 1: Data Privacy (Module Pattern)

```javascript
// Creating private state with closures
const User = (function() {
    // Private variables
    const users = new Map();
    let idCounter = 0;

    // Private function
    function generateId() {
        return ++idCounter;
    }

    // Public API
    return {
        create(name, email) {
            const id = generateId();
            users.set(id, { id, name, email, createdAt: new Date() });
            return id;
        },

        get(id) {
            const user = users.get(id);
            // Return copy to prevent mutation
            return user ? { ...user } : null;
        },

        update(id, updates) {
            const user = users.get(id);
            if (user) {
                Object.assign(user, updates);
                return true;
            }
            return false;
        },

        delete(id) {
            return users.delete(id);
        },

        getAll() {
            return Array.from(users.values()).map(u => ({ ...u }));
        }
    };
})();

// Usage
const userId = User.create('John', 'john@example.com');
console.log(User.get(userId));  // { id: 1, name: 'John', ... }
// users Map is not accessible!
```

### Case 2: Function Factory

```javascript
// Factory functions create specialized functions
function createMultiplier(factor) {
    // factor is "closed over" - preserved in the closure
    return function(number) {
        return number * factor;
    };
}

const double = createMultiplier(2);
const triple = createMultiplier(3);
const timesTen = createMultiplier(10);

console.log(double(5));     // 10
console.log(triple(5));     // 15
console.log(timesTen(5));   // 50

// Each function has its own factor value
console.log(double(3), triple(3), timesTen(3));  // 6, 3, 30

// More complex factory
function createValidator(rules) {
    return function(value) {
        const errors = [];

        for (const { test, message } of rules) {
            if (!test(value)) {
                errors.push(message);
            }
        }

        return {
            valid: errors.length === 0,
            errors
        };
    };
}

const emailValidator = createValidator([
    { test: v => v.includes('@'), message: 'Missing @' },
    { test: v => v.includes('.'), message: 'Missing .' }
]);

console.log(emailValidator('test@email.com'));  // { valid: true, errors: [] }
console.log(emailValidator('test'));             // { valid: false, errors: [...] }
```

### Case 3: Partial Application and Currying

```javascript
// Currying: transforming function with multiple args into chained functions
function curriedAdd(a) {
    return function(b) {
        return function(c) {
            return a + b + c;
        };
    };
}

const add = curriedAdd(1)(2)(3);  // 6

// More readable version
const add = a => b => c => a + b + c;

// Partial application: pre-fill some arguments
function partial(fn, ...presetArgs) {
    return function(...laterArgs) {
        return fn(...presetArgs, ...laterArgs);
    };
}

function greet(greeting, name, punctuation) {
    return `${greeting}, ${name}${punctuation}`;
}

const sayHello = partial(greet, 'Hello');
const sayGoodbye = partial(greet, 'Goodbye');

console.log(sayHello('John', '!'));    // Hello, John!
console.log(sayGoodbye('Jane', '?')); // Goodbye, Jane?

// Practical use: create event handlers with pre-filled data
function createEventHandler(eventName, handler) {
    return function(event) {
        console.log(`${eventName} triggered`);
        handler(event);
    };
}

const onClick = createEventHandler('click');
const onSubmit = createEventHandler('submit');

// Both preserve their eventName argument!
```

### Case 4: Async Closures with Loop

```javascript
// Classic closure problem in loops
for (var i = 0; i < 3; i++) {
    setTimeout(() => console.log(i), 100);
}
// Output: 3, 3, 3  (var is function-scoped, not block-scoped)

// Solution 1: Use let (block-scoped)
for (let i = 0; i < 3; i++) {
    setTimeout(() => console.log(i), 100);
}
// Output: 0, 1, 2

// Solution 2: Use IIFE to capture value
for (var i = 0; i < 3; i++) {
    (function(index) {
        setTimeout(() => console.log(index), 100);
    })(i);
}
// Output: 0, 1, 2

// Solution 3: Use forEach with closure
[0, 1, 2].forEach(function(index) {
    setTimeout(() => console.log(index), 100);
});
// Output: 0, 1, 2

// More complex async closure - maintaining state
function createAsyncProcessor(items) {
    let index = 0;

    return {
        async next() {
            if (index >= items.length) {
                return { done: true };
            }

            const item = items[index++];
            // Simulate async work
            const result = await processItem(item);

            return { done: false, value: result };
        }
    };
}

async function processItem(item) {
    return new Promise(resolve =>
        setTimeout(() => resolve(`Processed ${item}`), 100)
    );
}

// Usage
const processor = createAsyncProcessor(['a', 'b', 'c']);
console.log(await processor.next());  // { done: false, value: 'Processed a' }
console.log(await processor.next());  // { done: false, value: 'Processed b' }
```

### Case 5: Memoization

```javascript
// Caching expensive function results
function memoize(fn) {
    const cache = new Map();

    return function(...args) {
        const key = JSON.stringify(args);

        if (cache.has(key)) {
            return cache.get(key);
        }

        const result = fn.apply(this, args);
        cache.set(key, result);
        return result;
    };
}

// Example: Expensive Fibonacci
function fibonacci(n) {
    if (n <= 1) return n;
    return fibonacci(n - 1) + fibonacci(n - 2);
}

// Without memoization: O(2^n) - extremely slow for n > 40
// With memoization: O(n) - much faster!

const memoizedFibonacci = memoize(function(n) {
    if (n <= 1) return n;
    return memoizedFibonacci(n - 1) + memoizedFibonacci(n - 2);
});

console.log(memoizedFibonacci(50));  // Fast, cached results!

// Memoization with closure and timestamp for TTL
function memoizeWithTTL(ttl) {
    const cache = new Map();

    return function(fn) {
        return function(...args) {
            const key = JSON.stringify(args);
            const cached = cache.get(key);

            if (cached && Date.now() - cached.timestamp < ttl) {
                return cached.value;
            }

            const result = fn.apply(this, args);
            cache.set(key, { value: result, timestamp: Date.now() });
            return result;
        };
    };
}

const memoize5min = memoizeWithTTL(5 * 60 * 1000);
const fetchWithCache = memoize5min(fetch);
```

## Trade-offs: Closures vs Classes

### When to Use Closures

```javascript
// Closures are great for:
// 1. Simple data privacy
// 2. Function factories
// 3. Functional patterns (currying, partial application)

// Closure pattern
function createCounter() {
    let count = 0;
    return () => ++count;
}

// Class pattern
class Counter {
    constructor() {
        this.count = 0;
    }
    increment() {
        return ++this.count;
    }
}

// Closures are more lightweight - no 'new' keyword needed
const counter1 = createCounter();
const counter2 = createCounter();

console.log(counter1(), counter1());  // 1, 2
console.log(counter2(), counter2());  // 1, 2  (separate instances!)
```

### When Classes Might Be Better

```javascript
// Classes are better for:
// 1. Complex objects with many methods
// 2. When you need inheritance
// 3. When you need 'this' context in methods
// 4. When using 'instanceof' checks

class Animal {
    constructor(name) {
        this.name = name;
    }

    speak() {
        return `${this.name} makes a sound`;
    }
}

class Dog extends Animal {
    speak() {
        return `${this.name} barks`;
    }
}

const dog = new Dog('Rex');
console.log(dog.speak());  // Rex barks
console.log(dog instanceof Animal);  // true

// This is harder with closures due to prototype chain
```

## Production Considerations

### 1. Memory Leaks with Closures

```javascript
// Problem: Closures can retain references, causing memory leaks
function attachHandler(element) {
    const data = fetchDataForElement(element.id);  // Large object

    element.addEventListener('click', function() {
        // This closure retains reference to 'data'!
        console.log(data);
    });

    // If element is removed but listener isn't removed, data can't be GC'd
}

// Solution: Clean up listeners
function attachHandler(element) {
    const data = fetchDataForElement(element.id);

    const handler = () => console.log(data);
    element.addEventListener('click', handler);

    // Return cleanup function
    return function() {
        element.removeEventListener('click', handler);
    };
}

const cleanup = attachHandler(document.getElementById('btn'));
// When done: cleanup();  // Removes listener, data can be GC'd
```

### 2. Performance: Too Many Closures

```javascript
// Problem: Creating many closures can impact performance
function processItems(items) {
    return items.map(function(item) {
        // Each map creates a new function with closure scope
        return {
            id: item.id,
            display: function() { return item.name; }  // Another closure!
        };
    });
}

// Solution: Avoid nested closures, use bind or methods
function processItemsOptimized(items) {
    return items.map(item => ({
        id: item.id,
        name: item.name  // Access directly instead of function
    }));
}

// Or use method references
class ItemProcessor {
    constructor(item) {
        this.item = item;
    }

    getId() { return this.item.id; }
    getName() { return this.item.name; }
}

function processItemsClass(items) {
    return items.map(item => new ItemProcessor(item));
}
```

### 3. Debugging Closures

```javascript
// Closures can be hard to debug - add naming for clarity
const createHandler = (function createHandlerFactory(name) {  // Named function
    return function handleClick(event) {  // Named callback
        console.log(`Handler: ${name}`, event);
    };
})('my-button');


// In DevTools, closures show in Scope section
function outer() {
    const outerVar = 'outer';
    return function inner() {
        // debugger;  // Check 'Scope' panel in DevTools
        console.log(outerVar);
    };
}
```

## Common Mistakes

### Mistake 1: Not Understanding What Variables Are Captured

```javascript
// WRONG: Expecting closure to capture current value
function createButtons() {
    for (var i = 1; i <= 3; i++) {
        document.querySelector('body').innerHTML +=
            `<button id="btn${i}">Button ${i}</button>`;

        document.getElementById(`btn${i}`).addEventListener('click', function() {
            alert('Clicked button ' + i);  // Always shows '4'!
        });
    }
}

// CORRECT: Capture the value at creation time
function createButtonsFixed() {
    for (let i = 1; i <= 3; i++) {  // 'let' creates block scope
        document.getElementById(`btn${i}`).addEventListener('click', function() {
            alert('Clicked button ' + i);  // Works!
        });
    }
}

// Or use IIFE
function createButtonsIIFE() {
    for (var i = 1; i <= 3; i++) {
        (function(btnNum) {
            document.getElementById(`btn${btnNum}`).addEventListener('click', function() {
                alert('Clicked button ' + btnNum);  // Captured!
            });
        })(i);
    }
}
```

### Mistake 2: Modifying Closed Variables

```javascript
// WRONG: Mutating closure variable unexpectedly
function createCounter() {
    let count = 0;

    return {
        getCount() { return count; },
        increment() { return ++count; },
        decrement() { return --count; },
        reset() { count = 0; }  // Direct mutation - can cause bugs
    };
}

// Be aware: every reference to count points to SAME variable
const counter = createCounter();
counter.increment();
counter.increment();
console.log(counter.getCount());  // 2
counter.reset();  // Reset to 0

// Better: Return new instances instead of mutating
function createImmutableCounter() {
    let count = 0;

    return {
        getCount() { return count; },
        increment() {
            return createImmutableCounter(count + 1);
        }
    };
}
```

### Mistake 3: Forgetting 'this' in Closures

```javascript
// WRONG: Losing 'this' context
const person = {
    name: 'John',
    greet() {
        setTimeout(function() {
            console.log(this.name);  // undefined - 'this' is not person!
        }, 1000);
    }
};

// CORRECT: Preserve 'this'
const personFixed = {
    name: 'John',
    greet() {
        setTimeout(() => {
            console.log(this.name);  // 'John' - arrow function
        }, 1000);
    }
};

// Or use bind
const personBound = {
    name: 'John',
    greet() {
        setTimeout(function() {
            console.log(this.name);
        }.bind(this), 1000);
    }
};

// Or store reference
const personStored = {
    name: 'John',
    greet() {
        const self = this;
        setTimeout(function() {
            console.log(self.name);  // 'John'
        }, 1000);
    }
};
```

### Mistake 4: Creating Functions Inside Loops

```javascript
// WRONG: Creating functions in loop
const funcs = [];
for (var i = 0; i < 3; i++) {
    funcs.push(function() { return i; });  // All functions reference same i!
}
console.log(funcs[0](), funcs[1](), funcs[2]());  // 3, 3, 3

// CORRECT: Capture value
const funcsFixed = [];
for (let i = 0; i < 3; i++) {
    funcsFixed.push(function() { return i; });
}
console.log(funcsFixed[0](), funcsFixed[1](), funcsFixed[2]());  // 0, 1, 2

// Or with index
const funcsWithIIFE = [];
for (var i = 0; i < 3; i++) {
    funcsWithIIFE.push((function(index) {
        return function() { return index; };
    })(i));
}
```

### Mistake 5: Using Closures When They're Not Needed

```javascript
// WRONG: Overcomplication
function getData(id) {
    const cache = {};

    return function fetchFromCache(key) {
        if (cache[key]) return cache[key];

        const data = expensiveOperation(key);
        cache[key] = data;
        return data;
    };
}

// CORRECT: Use built-in features or simpler patterns
function getDataSimple(id) {
    const cache = new Map();

    return (key) => {
        if (!cache.has(key)) {
            cache.set(key, expensiveOperation(key));
        }
        return cache.get(key);
    };
}

// Even simpler: use Map directly
const dataCache = new Map();
function getData(id) {
    if (!dataCache.has(id)) {
        dataCache.set(id, expensiveOperation(id));
    }
    return dataCache.get(id);
}
```

## Summary

Closures are a fundamental JavaScript concept:

1. **Definition**: A function that accesses variables from its outer scope
2. **Creation**: Every function creates a closure at definition time
3. **Use Cases**: Data privacy, function factories, currying, async state
4. **Memory**: Closures retain their outer scope—be careful of memory leaks

Key insights:
- Use closures for simple data privacy
- Avoid creating closures in loops without proper scoping
- Use arrow functions to preserve `this` context
- Be aware of memory implications with long-lived closures

Closures are what make JavaScript powerful for functional programming patterns.
