# JavaScript Event Loop: The Complete Guide

## Overview

JavaScript is single-threaded, meaning it can only execute one piece of code at a time. Yet, we build asynchronous applications that handle network requests, user interactions, and timers without freezing. The event loop is the mechanism that makes this possible.

Understanding how the event loop works is essential for debugging async issues, writing performant code, and understanding why certain patterns behave unexpectedly.

---

## How the Event Loop Works Internally

### The JavaScript Runtime Architecture

When JavaScript executes code, it uses several internal components:

```
┌─────────────────────────────────────────────────────────────────┐
│                      JavaScript Runtime                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌─────────────┐     ┌─────────────┐     ┌─────────────┐       │
│  │  Call Stack │     │ Web APIs    │     │  Callback   │       │
│  │             │     │             │     │   Queue     │       │
│  │  main()     │     │ setTimeout  │     │             │       │
│  │  └─ func()  │     │ fetch       │     │  ┌───────┐ │       │
│  │     └─ log  │     │ DOM events  │     │  │ fn()  │ │       │
│  └─────────────┘     └─────────────┘     │  │ cb()  │ │       │
│        │                  │              │  └───────┘ │       │
│        │                  │              └─────────────┘       │
│        │                  │                    │               │
│        ▼                  ▼                    ▼               │
│  ┌──────────────────────────────────────────────────────┐      │
│  │                    Event Loop                         │      │
│  └──────────────────────────────────────────────────────┘      │
│                              │                                   │
│                              ▼                                   │
│  ┌──────────────────────────────────────────────────────┐      │
│  │                   Microtask Queue                    │      │
│  │        (Promise callbacks, MutationObserver)         │      │
│  └──────────────────────────────────────────────────────┘      │
└─────────────────────────────────────────────────────────────────┘
```

### Call Stack

The call stack is where function calls are tracked. When a function is called, it's pushed onto the stack. When it returns, it's popped off.

```javascript
function greet() {
    console.log("Hello!");
}

function sayGreet() {
    greet();  // push greet onto stack
}

sayGreet();  // push sayGreet onto stack

// Execution order:
// 1. push sayGreet
// 2. push greet
// 3. pop greet (returns)
// 4. pop sayGreet (returns)
```

### The Event Loop Cycle

The event loop continuously checks if the call stack is empty. If it is, it moves callbacks from the task queue to the call stack:

```javascript
console.log("1: Start");

setTimeout(() => {
    console.log("2: Timeout callback");
}, 0);

Promise.resolve().then(() => {
    console.log("3: Promise resolved");
});

console.log("4: End");

// Output:
// 1: Start
// 4: End
// 3: Promise resolved
// 2: Timeout callback
```

The key insight: microtasks (Promises) run before macrotasks (setTimeout), even when the Promise resolves immediately.

## Real-World JavaScript Use Cases

### Case 1: Understanding Promise Execution Order

```javascript
// Multiple promises with different settle times
async function executionOrder() {
    console.log("1. Start");

    // Promise 1: Immediately resolves
    Promise.resolve().then(() => console.log("2. Promise 1"));

    // Promise 2: Resolves after timeout
    new Promise(resolve => setTimeout(resolve, 0))
        .then(() => console.log("3. Timeout Promise"));

    // Promise 3: Resolves immediately with chaining
    Promise.resolve()
        .then(() => console.log("4. First chain"))
        .then(() => console.log("5. Second chain"));

    // setTimeout with delay 0
    setTimeout(() => console.log("6. SetTimeout"), 0);

    console.log("7. End");
}

executionOrder();

// Output:
// 1. Start
// 7. End
// 2. Promise 1
// 4. First chain
// 5. Second chain
// 3. Timeout Promise (microtask after current sync code)
// 6. SetTimeout (macrotask)
```

### Case 2: Blocking the Main Thread

```javascript
// This function blocks the main thread
function heavyComputation() {
    let result = 0;
    for (let i = 0; i < 1000000000; i++) {
        result += i;
    }
    return result;
}

// The UI freezes while this runs
// document.querySelector('button').disabled = true;

// Solution: Use Web Workers or break into chunks
function chunkedComputation() {
    let result = 0;
    let i = 0;
    const chunkSize = 1000000;

    function processChunk() {
        const end = Math.min(i + chunkSize, 1000000000);
        for (; i < end; i++) {
            result += i;
        }

        if (i < 1000000000) {
            // Schedule next chunk on next tick
            setTimeout(processChunk, 0);
        } else {
            console.log("Done:", result);
        }
    }

    processChunk();
}

// UI stays responsive while computing
```

### Case 3: QueueMicrotask for Priority

```javascript
// queueMicrotask schedules before setTimeout but after Promises
function demonstrateMicrotask() {
    console.log("1. Sync start");

    queueMicrotask(() => console.log("2. queueMicrotask"));

    Promise.resolve().then(() => console.log("3. Promise"));

    setTimeout(() => console.log("4. setTimeout"), 0);

    console.log("5. Sync end");
}

demonstrateMicrotask();

// Output:
// 1. Sync start
// 5. Sync end
// 2. queueMicrotask
// 3. Promise
// 4. setTimeout

// queueMicrotask is useful for:
function asyncTask() {
    // Ensuring synchronous code runs first
    queueMicrotask(() => {
        // This runs after current synchronous code
        // but before other async operations
    });
}
```

### Case 4: Understanding async/await Under the Hood

```javascript
// async/await is syntactic sugar for Promises
// Each await is essentially a .then() on a Promise

async function example() {
    const a = await Promise.resolve(1);
    const b = await Promise.resolve(2);
    console.log(a + b);
}

// This compiles to something like:
function exampleCompiled() {
    return Promise.resolve(1).then(a => {
        return Promise.resolve(2).then(b => {
            console.log(a + b);
        });
    });
}

// Important: await returns microtask, not macrotask
async function timing() {
    console.log("1. Before await");

    await Promise.resolve();
    console.log("2. After await"); // Still runs synchronously after!

    setTimeout(() => console.log("3. setTimeout"), 0);
    console.log("4. After setTimeout");
}

timing();

// Output:
// 1. Before await
// 2. After await
// 4. After setTimeout
// 3. setTimeout

// The await doesn't create a new macrotask - it continues in the same tick
```

## Trade-offs: Microtasks vs Macrotasks

### When to Use Which

| Operation | Type | Use Case |
|-----------|------|----------|
| `Promise.then()` | Microtask | Dependent async operations |
| `setTimeout` | Macrotask | Delaying execution |
| `setInterval` | Macrotest | Repeating intervals |
| `queueMicrotask` | Microtask | Highest priority async |
| `requestAnimationFrame` | Macrotask | Animation frames |

### Performance Implications

```javascript
// Problem: Too many microtasks can block rendering
function problematicLoop() {
    for (let i = 0; i < 10000; i++) {
        Promise.resolve().then(() => {
            // Each promise creates a microtask
            // Browser can't paint between them
            updateUI(i);
        });
    }
}

// Solution: Batch with setTimeout or use requestIdleCallback
function optimizedLoop() {
    let i = 0;

    function batch() {
        const batchSize = 100;
        for (let j = 0; j < batchSize && i < 10000; j++, i++) {
            updateUI(i);
        }

        if (i < 10000) {
            setTimeout(batch, 0); // Let browser breathe
        }
    }

    batch();
}
```

## Production Considerations

### 1. Debugging Async Issues

```javascript
// Enable 'Async' stack traces in Chrome DevTools
// This shows async call chains in the console

// Common async debugging pattern
async function debugAsync() {
    try {
        const data = await fetch('/api/data');
        const json = await data.json();
        console.log(json);
    } catch (error) {
        console.error('Error:', error);
        // Add breakpoints in catch blocks
    }
}
```

### 2. Avoiding Stack Overflow

```javascript
// Recursive functions can overflow the call stack
function recursiveSum(n) {
    if (n <= 0) return 0;
    return n + recursiveSum(n - 1);
}

// Solution: Use iteration or chunked recursion
function iterativeSum(n) {
    let sum = 0;
    for (let i = 1; i <= n; i++) {
        sum += i;
    }
    return sum;
}

function chunkedSum(n) {
    let sum = 0;
    function addNext(current) {
        const end = Math.min(current + 1000, n);
        for (let i = current; i < end; i++) {
            sum += i + 1;
        }
        if (end < n) {
            setTimeout(() => addNext(end), 0);
        }
    }
    addNext(1);
}
```

### 3. Node.js vs Browser Event Loop

```javascript
// Node.js has additional phases not present in browsers
const fs = require('fs');

// In Node.js, there are multiple phases:
// ┌───────────────────────┐
// │        poll          │ <- I/O callbacks, async I/O
// │        check          │ <- setImmediate callbacks
// │       close           │ <- socket.on('close')
// └───────────────────────┘

// setTimeout vs setImmediate
setTimeout(() => console.log('timeout'), 0);
setImmediate(() => console.log('immediate'));

// In main module: timeout may execute first
// In I/O callback: setImmediate always executes first
```

## Common Mistakes

### Mistake 1: Assuming setTimeout(fn, 0) Executes Immediately

```javascript
// WRONG: Thinking setTimeout(fn, 0) executes in the next tick
setTimeout(() => console.log('1'), 0);
console.log('2');
// Output: 2, 1  (correct order, but NOT immediate)

// The '0' means minimum delay, not instant execution
// It goes to macrotask queue, runs after microtasks

// CORRECT: Use queueMicrotask for immediate (but still async)
queueMicrotask(() => console.log('microtask'));
console.log('sync');
// Output: sync, microtask
```

### Mistake 2: Not Understanding await Blocking

```javascript
// WRONG: await in a loop - sequential execution
async function badPattern(urls) {
    const results = [];
    for (const url of urls) {
        const response = await fetch(url);  // Waits for each!
        results.push(await response.json());
    }
    return results;
}

// CORRECT: Parallel execution with Promise.all
async function goodPattern(urls) {
    const promises = urls.map(url => fetch(url).then(r => r.json()));
    return Promise.all(promises);
}

// For sequential needed:
async function sequentialPattern(urls) {
    const results = [];
    for (const url of urls) {
        results.push(await fetchSequentially(url));
    }
    return results;
}
```

### Mistake 3: Forgetting Promise Chain Async

```javascript
// WRONG: Thinking synchronous code runs before promise callbacks
async function syncFirst() {
    const result = await Promise.resolve(console.log('1'));
    console.log('2');  // Runs after the await!
}
syncFirst();
console.log('3');
// Output: 1, 3, 2

// The await pauses the function execution until resolved
// Then continues in the same tick's microtask phase

// CORRECT: Understanding the timing
async function understandTiming() {
    console.log('1');  // Sync

    await console.log('2');  // Pauses here

    console.log('3');  // Runs as continuation (microtask)

    setTimeout(() => console.log('4'), 0);  // Macrotask
}
console.log('5');
understandTiming();
console.log('6');
// Output: 5, 1, 2, 3, 6, 4
```

### Mistake 4: Mixing Microtasks and UI Updates

```javascript
// WRONG: Updating UI in microtask - causes jank
function updateInLoop() {
    for (let i = 0; i < 1000; i++) {
        Promise.resolve().then(() => {
            document.body.innerHTML = i;  // Causes reflow!
        });
    }
}

// Browser cannot paint between microtasks
// UI freezes until all microtasks complete

// CORRECT: Use requestAnimationFrame for visual updates
function updateWithRAF() {
    let i = 0;
    function frame() {
        if (i < 1000) {
            document.body.innerHTML = i;
            i++;
            requestAnimationFrame(frame);
        }
    }
    requestAnimationFrame(frame);
}
```

### Mistake 5: Not Handling Promise Rejection

```javascript
// WRONG: Unhandled promise rejection
fetch('/api/data')
    .then(data => process(data));
// If fetch fails, unhandled rejection warning!

// CORRECT: Always handle rejections
fetch('/api/data')
    .then(data => process(data))
    .catch(error => console.error('Fetch failed:', error));

// Or use try/catch with async/await
async function safeFetch() {
    try {
        const response = await fetch('/api/data');
        return await response.json();
    } catch (error) {
        console.error('Error:', error);
        return null;
    }
}

// For global unhandled rejection monitoring
window.addEventListener('unhandledrejection', event => {
    console.error('Unhandled rejection:', event.reason);
});
```

## Summary

The event loop is the heart of JavaScript's async behavior:

1. **Call Stack**: Executes synchronous code, one function at a time
2. **Microtask Queue**: Processes Promises before macrotasks
3. **Macrotask Queue**: Processes setTimeout, setInterval, I/O
4. **Event Loop**: Moves callbacks from queues to stack when empty

Key takeaways:
- Microtasks run to completion before macrotasks
- async/await pauses function execution, not entire program
- Use `queueMicrotask` for priority async operations
- Use `requestAnimationFrame` for UI updates in loops

Understanding the event loop is essential for writing predictable, performant JavaScript.
