# Load Balancing

## Overview

Load balancing is a critical infrastructure component that distributes incoming network traffic across multiple servers to ensure no single server becomes overwhelmed, thereby improving application availability, reliability, and performance. In modern distributed systems, load balancers act as the first line of defense against traffic spikes and serve as the traffic cop that directs requests to the most appropriate backend resources.

This guide explores load balancing fundamentals, the various types of load balancers available, algorithms for traffic distribution, health monitoring strategies, and practical implementation considerations for building resilient systems.

## Problem Statement

Modern applications face significant challenges that load balancing addresses:

**Traffic Concentration**: Without load balancing, a single server handling all incoming requests becomes a bottleneck and single point of failure. When traffic increases beyond a server's capacity, users experience slow responses or complete service outages.

**Resource Utilization**: Inefficient distribution leads to some servers being overutilized while others sit idle. This waste of resources increases costs while degrading user experience.

**High Availability Requirements**: Users expect applications to be available 24/7. Load balancing enables rolling updates, automatic failover, and maintenance without downtime.

**Geographic Distribution**: Global applications need to route users to the nearest healthy server to minimize latency and provide consistent experiences across regions.

## Types of Load Balancers

### Layer 4 (Transport Layer) Load Balancing

Layer 4 load balancers operate at the TCP/UDP level, making routing decisions based on IP address and port number without inspecting the actual content of the request. This approach offers several advantages:

**Advantages:**
- Minimal latency as packets are forwarded without content inspection
- Higher throughput since processing overhead is minimal
- Simpler implementation with less computational overhead
- Suitable for any TCP/UDP-based protocol

**Disadvantages:**
- Cannot make routing decisions based on application-layer content
- Limited optimization capabilities for specific protocols
- No session persistence based on application state

Common use cases include database connections, streaming services, and scenarios where raw throughput is critical.

### Layer 7 (Application Layer) Load Balancing

Layer 7 load balancers operate at the HTTP/HTTPS level, inspecting request content including headers, cookies, and URLs to make sophisticated routing decisions:

**Advantages:**
- Can route based on URL path, headers, or content
- Enables advanced features like path-based routing, sticky sessions, and request/response modification
- Better optimization for HTTP-based workloads
- Support for SSL/TLS termination and load balancing

**Disadvantages:**
- Higher latency due to content inspection
- More computational overhead than Layer 4
- Memory intensive for handling large volumes of connections

Layer 7 load balancing is ideal for web applications, API gateways, and microservices architectures where intelligent routing is required.

### Global Server Load Balancing (GSLB)

GSLB extends load balancing across multiple geographic regions, directing users to the nearest or most appropriate data center:

**DNS-based GSLB**: Responds to DNS queries with IP addresses of the closest or best-performing data center.

**Anycast-based GSLB**: Uses BGP anycast to advertise the same IP address from multiple locations, naturally routing users to the nearest node.

**Hybrid GSLB**: Combines health monitoring, user location, and real-time performance data to make routing decisions.

## Load Balancing Algorithms

### Round Robin

The simplest algorithm that distributes requests evenly across all servers in sequence. Each incoming request is routed to the next server in the list, cycling back to the beginning when the end is reached.

**Best for**: Homogeneous server environments with similar capacities and roughly equal request processing times.

**Limitations**: Doesn't account for varying server capacities or current load.

### Least Connections

Routes new requests to the server with the fewest active connections. This algorithm is dynamic and adapts to varying request processing times:

**Best for**: Applications with varied request processing times, such as those involving database queries or external API calls.

**How it works**: Maintains a count of active connections for each server and routes to the one with the lowest count.

### Weighted Least Connections

An extension of least connections where servers are assigned weights based on their capacity. Higher-weighted servers receive more connections:

**Best for**: Heterogeneous server environments where different machines have varying capabilities.

**Implementation**: Server weight is considered alongside active connection count when making routing decisions.

### IP Hash

Routes requests to servers based on a hash of the client's IP address, ensuring the same client always reaches the same server:

**Best for**: Applications requiring session persistence when sticky sessions aren't available.

**Considerations**: Server additions or removals can cause significant redistribution of existing clients.

### Least Request Time

Routes to the server with the fastest response time, measured by either the least number of requests or shortest average response time:

**Best for**: Highly variable workloads where server performance fluctuates significantly.

**Adaptive nature**: Naturally balances load toward healthier, faster-responding servers.

### Consistent Hashing

Uses a hash ring to assign keys (requests) to servers, minimizing redistribution when servers are added or removed:

**Best for**: Distributed caching systems and scenarios where cache hit rates are critical.

**Advantages**: Reduces cache invalidation when scaling, as most existing mappings remain valid.

## Health Checks

Health checks are critical for ensuring traffic is only routed to functional servers. They operate at multiple levels:

### TCP Health Checks

Attempts TCP connection to verify the server is reachable at the network level. Simple and fast but doesn't verify application health.

### HTTP/HTTPS Health Checks

Sends HTTP requests to application endpoints to verify proper application function. Can check specific health indicators:

```yaml
healthcheck:
  interval: 10s
  timeout: 5s
  healthy_threshold: 2
  unhealthy_threshold: 3
  path: /health
  expected_status: 200
```

### Deep Health Checks

Performs actual functional testing, such as querying a database or calling a critical service:

```yaml
deep_health_check:
  type: custom
  script: /opt/healthcheck.sh
  interval: 30s
  timeout: 10s
```

### Health Check Best Practices

**Multiple checks**: Combine lightweight and heavyweight checks for comprehensive monitoring.

**Grace period**: Allow new servers time to warm up before including them in the pool.

**Failure thresholds**: Require consecutive failures before removing servers to avoid flapping.

**Probe diversity**: Use multiple probe locations to detect localized issues.

## SSL/TLS Termination

Many load balancers handle SSL/TLS termination to offload cryptographic processing from backend servers:

**Benefits**:
- Reduces computational load on application servers
- Centralizes certificate management
- Enables content inspection for Layer 7 load balancing
- Simplifies backend server configuration

**Considerations**:
- Increases load balancer CPU requirements
- Requires careful key management
- May have regulatory implications for certain deployments

## Sticky Sessions

Sticky sessions (session affinity) ensure a user's requests consistently reach the same server:

### Cookie-based Stickiness

Uses cookies to identify session affiliation. The load balancer reads or sets cookies to maintain session affinity:

```
Set-Cookie: JSESSIONID=abc123; Path=/; Sticky
```

### IP-based Stickiness

Maps client IPs to servers, useful when cookies can't be used:

**Limitations**: Not suitable for mobile networks or NAT'd traffic.

### Header-based Stickiness

Uses custom headers for session identification.

## Load Balancer Placement

### Edge Load Balancing

Places load balancers at the network edge, closest to users:

- Handles incoming traffic from the internet
- Performs DDoS protection and rate limiting
- Terminates SSL at the edge
- Routes to internal infrastructure

### Internal Load Balancing

Load balancers between application tiers:

- Distributes traffic among application servers
- Handles service-to-service communication
- Enables microservices load balancing

### Multi-tier Load Balancing

Combines edge and internal load balancing for comprehensive architecture:

```
Internet → Edge LB → App Servers → Internal LB → Database
```

## Implementation Considerations

### Selection Criteria

**Traffic type**: HTTP/HTTPS requests benefit from Layer 7; raw TCP/UDP benefits from Layer 4.

**Scale requirements**: Some load balancers scale better than others.

**Feature requirements**: SSL termination, persistent sessions, caching, etc.

**Cost structure**: Per-connection, per-bandwidth, or fixed pricing models.

### Scaling Strategies

**Horizontal scaling**: Add more load balancer instances behind a front-end load balancer.

**Vertical scaling**: Upgrade load balancer instance sizes.

**Connection draining**: Gracefully migrate connections during scaling.

### Monitoring and Observability

**Key metrics**:
- Request rate (requests per second)
- Latency percentiles (p50, p95, p99)
- Error rates
- Active connections
- Server health status
- Backlog queue depth

## Best Practices

1. **Always use health checks**: Enable comprehensive health monitoring to remove unhealthy servers automatically.

2. **Distribute across availability zones**: Deploy load balancers and backend servers across multiple availability zones for fault tolerance.

3. **Enable connection draining**: Gracefully handle server removal to prevent active request failures.

4. **Monitor load balancer health**: Load balancers themselves can fail; monitor them and have redundancy.

5. **Use appropriate algorithms**: Select algorithms matching your workload characteristics—least connections for variable workloads.

6. **Implement SSL offloading**: Terminate SSL at the load balancer to reduce backend computational burden.

7. **Plan for scaling**: Design for scale from the beginning; consider capacity limits of your load balancer solution.

## Common Mistakes

1. **Single point of failure**: Deploying a single load balancer without redundancy.

2. **Ignoring health check tuning**: Using default health checks that are too aggressive or not aggressive enough.

3. **Inadequate connection limits**: Not understanding connection limits of load balancer or underlying infrastructure.

4. **Sticky session overuse**: Over-relying on sticky sessions complicates scaling and failover.

5. **Neglecting SSL/TLS costs**: Underestimating CPU requirements for high SSL traffic volumes.

6. **Ignoring geographic distribution**: Not planning for multi-region deployments early enough.

## Summary

Load balancing is foundational to building scalable, available systems. The right load balancing architecture depends on your specific requirements—traffic type, scale, geographic distribution, and feature needs. Layer 4 load balancing provides simplicity and performance, while Layer 7 offers sophisticated routing capabilities. The key is selecting appropriate algorithms, implementing robust health checks, and planning for horizontal scaling from the start.

Modern applications often use multiple layers of load balancing—at the edge for internet traffic, internally between services, and at each tier of the architecture. This multi-tier approach provides flexibility, improves performance, and enables the resilience that modern users expect.
