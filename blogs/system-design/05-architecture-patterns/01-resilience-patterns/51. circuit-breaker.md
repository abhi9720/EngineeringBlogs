# Circuit Breaker Pattern

## Overview

The circuit breaker pattern is a critical resilience pattern that prevents cascading failures in distributed systems. Just like electrical circuit breakers prevent fires by stopping excessive current, software circuit breakers prevent system failures by stopping calls to failing services and allowing them time to recover.

This guide explores circuit breaker fundamentals, state machine implementation, configuration strategies, and practical patterns for building resilient microservices.

## Problem Statement

Distributed systems face numerous challenges:

**Cascading Failures**: One failing service can cause others to fail as they wait for responses, exhausting resources.

**Slow Recovery**: Failing services often need time to recover but requests keep coming, preventing recovery.

**Poor UX**: Users experience timeouts and errors when services are temporarily unavailable.

**Resource Exhaustion**: Connection pools, threads, and memory get exhausted waiting on failing services.

### State Transitions

| State | Description |
|-------|-------------|
| **Closed** | Normal operation; failures are counted |
| **Open** | Calls fail immediately; service is considered unavailable |
| **Half-Open** | Limited testing calls are allowed |

## Implementation

### Basic Circuit Breaker

```java
public class CircuitBreaker {
    private final String serviceName;
    private final CircuitBreakerConfig config;
    
    private State state = State.CLOSED;
    private int failureCount = 0;
    private int successCount = 0;
    private Instant lastFailureTime;
    
    public enum State { CLOSED, OPEN, HALF_OPEN }
    
    public <T> T execute(Supplier<T> operation) {
        switch (state) {
            case OPEN:
                if (shouldAttemptRecovery()) {
                    state = State.HALF_OPEN;
                } else {
                    throw new CircuitBreakerOpenException(serviceName);
                }
                break;
                
            case HALF_OPEN:
                // Allow limited requests
                break;
        }
        
        try {
            T result = operation.get();
            onSuccess();
            return result;
        } catch (Exception e) {
            onFailure();
            throw e;
        }
    }
    
    private void onSuccess() {
        failureCount = 0;
        
        if (state == State.HALF_OPEN) {
            state = State.CLOSED;
        }
    }
    
    private void onFailure() {
        failureCount++;
        
        if (failureCount >= config.getFailureThreshold()) {
            state = State.OPEN;
            lastFailureTime = Instant.now();
        }
    }
    
    private boolean shouldAttemptRecovery() {
        return lastFailureTime != null &&
            Duration.between(lastFailureTime, Instant.now())
                .compareTo(config.getRecoveryTimeout()) > 0;
    }
}
```

## CircuitBreaker with Resilience4j

Resilience4j is the most popular circuit breaker library:

### Configuration

```java
@Configuration
public class Resilience4jConfig {
    
    @Bean
    public CircuitBreakerRegistry circuitBreakerRegistry(CircuitBreakerConfig config) {
        CircuitBreakerRegistry registry = CircuitBreakerRegistry.of(config);
        
        registry.addConfiguration("default", Map.of(
            "failureRateThreshold", 50,
            "slowCallRateThreshold", 100,
            "slowCallDurationThreshold", Duration.ofSeconds(2),
            "waitDurationInOpenState", Duration.ofSeconds(30),
            "permittedNumberOfCallsInHalfOpenState", 3,
            "slidingWindowSize", 10,
            "minimumNumberOfCalls", 20
        ));
        
        return registry;
    }
}
```

### Usage

```java
@Service
public class ExternalServiceClient {
    
    @CircuitBreaker(name = "external-api", fallbackMethod = "fallback")
    public ExternalResponse callExternalApi(Request request) {
        return externalApiClient.call(request);
    }
    
    private ExternalResponse fallback(Request request, Exception e) {
        // Return cached data or default response
        return getCachedResponse(request.getId());
    }
}
```

## Fallback Strategies

### Cache Fallback

```java
public class CacheFallback {
    
    @CircuitBreaker(name = "userService")
    public User getUser(String userId) {
        return restTemplate.getForObject(url, User.class);
    }
    
    private User getUserFallback(String userId, CircuitBreakerException e) {
        // Try cache
        User cached = cache.get(userId);
        if (cached != null) {
            return cached;
        }
        
        // Return default
        return User.getDefault();
    }
}
```

### Default Value Fallback

```java
public class DefaultValueFallback {
    
    @CircuitBreaker(name = "inventory")
    public InventoryResult checkInventory(String productId) {
        return inventoryClient.check(productId);
    }
    
    private InventoryResult checkInventoryFallback(String productId, Exception e) {
        // Return "unknown" - safe default
        return InventoryResult.builder()
            .available(false)
            .reason("Service temporarily unavailable")
            .isStaleData(true)
            .build();
    }
}
```

### Queue for Later

```java
public class QueueFallback {
    
    @CircuitBreaker(name = "notification")
    public void sendNotification(Notification notification) {
        notificationClient.send(notification);
    }
    
    private void sendNotificationFallback(Notification notification, Exception e) {
        // Queue for retry
        retryQueue.add(notification);
        log.warn("Notification queued for retry: {}", notification.getId());
    }
}
```

### Health Indicator

```java
@Component
public class CircuitBreakerHealthIndicator implements ReactiveHealthIndicator {
    
    @Autowired
    private CircuitBreakerRegistry registry;
    
    @Override
    public Mono<Health> health() {
        Map<String, CircuitBreakerSummary> circuits = registry.getAllCircuitBreakers();
        
        boolean allClosed = circuits.values().stream()
            .allMatch(c -> c.getState() == CircuitBreaker.State.CLOSED);
        
        return Mono.just(Health.up()
            .withDetail("circuitBreakers", circuits)
            .status(allClosed ? HealthStatus.UP : HealthStatus.UNKNOWN)
            .build());
    }
}
```

## Advanced Patterns

### Bulkhead Integration

```java
public class CircuitBreakerWithBulkhead {
    
    @Bulkhead(name = "external-api")
    @CircuitBreaker(name = "external-api")
    public ExternalResponse callApi(Request request) {
        return externalClient.call(request);
    }
    
    // Bulkhead prevents resource exhaustion
    // Circuit breaker handles service failures
```

## Best Practices

1. **Use appropriate thresholds**: Tune based on your service SLAs.

2. **Implement fallbacks**: Always have a fallback strategy.

3. **Monitor states**: Track circuit breaker state changes.

4. **Log state transitions**: Important for debugging.

5. **Combine with timeouts**: Don't wait forever on slow services.

6. **Test circuit opening**: Include in your chaos testing.

7. **Graceful defaults**: Initial state should be CLOSED.

## Common Mistakes

1. **No fallback**: Leaving circuit breaker without fallback.

2. **Wrong thresholds**: Too aggressive or too lenient.

3. **No monitoring**: Not tracking circuit state.

4. **Ignoring half-open**: Not testing recovery properly.

5. **Catching exception incorrectly**: Should catch all exceptions.

## Summary

Circuit breakers are essential for resilient distributed systems. The pattern prevents cascading failures by stopping requests to failing services, allowing them time to recover. Key steps: implement proper state machine, configure thresholds based on SLAs, always provide fallbacks, and monitor state changes. Combined with timeouts, retries, and bulkheads, circuit breakers form a powerful resilience toolkit.
