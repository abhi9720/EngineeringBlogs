# File Storage Systems

## Overview

File storage systems provide traditional file-based access to storage, enabling applications to use familiar file operations (open, read, write, seek) for data storage. While object storage has become dominant for cloud applications, file storage systems remain critical for enterprise applications, legacy systems, and specific use cases requiring file semantics.

This guide explores file storage fundamentals, network file systems (NFS, SMB), distributed file systems, and practical implementation strategies.

## Problem Statement

Applications require file-based access:

**Legacy Applications**: Existing apps designed for file systems.

**Shared File Access**: Multiple servers need shared access to same files.

**Operating System Integration**: File systems integrate with OS tools.

**Standard Protocols**: NFS, SMB are well-understood protocols.

**No Database Needed**: Simpler than databases for some use cases.

## Storage Types Comparison

| Aspect | File Storage | Block Storage | Object Storage |
|--------|-------------|---------------|----------------|
| **Access** | POSIX, SMB | Block devices | REST API |
| **Use Case** | Shared files, legacy | Databases, VMs | Cloud apps |
| **Scale** | Limited | Medium | Unlimited |
| **Cost** | Medium | Higher | Lower |
| **Management** | More | Medium | Less |

## NFS (Network File System)

Network file system provides file access over network:

### NFS Server Setup (Linux)

```bash
# Install NFS server
sudo apt-get install nfs-kernel-server

# Create export directory
sudo mkdir -p /exports/data
sudo chmod 777 /exports/data

# Configure exports
echo "/exports/data *(rw,sync,no_subtree_check,no_root_squash)" | sudo tee /etc/exports

# Export and start
sudo exportfs -a
sudo systemctl start nfs-kernel-server
```

### /etc/exports Configuration

```
# Export options
/exports/data        client1(rw,sync)                               # Single client
/exports/data        *.example.com(rw,sync)                        # By hostname  
/exports/data        192.168.1.0/24(rw,sync,no_subtree_check)     # By subnet
/exports/readonly   *(ro,sync,no_subtree_check)                   # Read-only
/exports/secure     192.168.1.10(rw,sync,no_subtree_check,root_squash)  # Root squash
```

### NFS Mount Options

```bash
# Basic mount
mount -t nfs server1:/exports/data /mnt/data

# With options
mount -t nfs -o rw,hard,intr,rsize=8192,wsize=8192 server1:/exports/data /mnt/data

# In /etc/fstab
server1:/exports/data    /mnt/data    nfs    rw,hard,intr,_rnetdev    0 0
```

### NFS Client Java Implementation

```java
public class NFSClientService {
    
    public void mount(String server, String export, String mountPoint) {
        try {
            ProcessBuilder pb = new ProcessBuilder(
                "mount", "-t", "nfs", 
                server + ":" + export, mountPoint
            );
            pb.inheritIO();
            Process process = pb.start();
            process.waitFor();
            
            if (process.exitValue() != 0) {
                throw new RuntimeException("Mount failed");
            }
        } catch (Exception e) {
            throw new RuntimeException("Mount failed", e);
        }
    }
    
    public File[] listFiles(String mountPoint) {
        File directory = new File(mountPoint);
        return directory.listFiles();
    }
    
    public void copy(String sourcePath, String destNfsPath) throws IOException {
        Files.copy(
            Paths.get(sourcePath), 
            Paths.get(destNfsPath),
            StandardCopyOption.REPLACE_EXISTING
        );
    }
}
```

## SMB/CIFS

Server Message Block is Windows-standard file sharing:

### SMB Server (Samba)

```bash
# Install Samba
sudo apt-get install samba

# Configure smb.conf
sudo nano /etc/samba/smb.conf
```

### smb.conf

```ini
[global]
   workgroup = WORKGROUP
   server string = %h server
   log file = /var/log/samba/log.%m
   max log size = 1000
   
[shared]
   path = /exports/shared
   valid users = @users
   writable = yes
   browsable = yes
   create mask = 0775

[readonly]
   path = /exports/readonly
   public = yes
   writable = no
   create mask = 0555
```

### SMB Client (Linux/Windows)

```java
public class SMBClientService {
    
    // Using JCIFS library
    public void connect() throws IOException {
        NtlmPasswordAuthentication auth = new NtlmPasswordAuthentication(
            "", "username", "password"
        );
        
        SmbFile smbFile = new SmbFile(
            "smb://server/share/path", auth
        );
    }
    
    public void copyFile(String smbPath, String localPath) 
            throws IOException {
        SmbFile remoteFile = new SmbFile(smbPath);
        
        try (InputStream in = remoteFile.getInputStream();
             OutputStream out = new FileOutputStream(localPath)) {
            
            byte[] buffer = new byte[8192];
            int read;
            
            while ((read = in.read(buffer)) != -1) {
                out.write(buffer, 0, read);
            }
        }
    }
}
```

## Distributed File Systems

### GlusterFS

```bash
# Install GlusterFS
sudo apt-get install glusterfs-server glusterfs-client

# Create volume
gluster volume create myvolume replica 2 \
    server1:/bricks/brick1 \
    server2:/bricks/brick1 \
    server3:/bricks/brick1 \
    server4:/bricks/brick1
    
# Start volume
gluster volume start myvolume

# Mount
mount -t glusterfs server1:/myvolume /mnt/gluster
```

### GlusterFS Java Client

```java
public class GlusterFSClient {
    
    public void write(String path, byte[] data) throws IOException {
        // GlusterFS mounts as regular file system
        // Just use Java NIO
        Path filePath = Paths.get("/mnt/gluster/" + path);
        
        Files.write(filePath, data);
    }
    
    public byte[] read(String path) throws IOException {
        Path filePath = Paths.get("/mnt/gluster/" + path);
        
        return Files.readAllBytes(filePath);
    }
}
```

### CephFS

```bash
# Mount CephFS using kernel client
mount -t ceph ceph-mon1:6789:/ /mnt/ceph -o name=admin,secret=...

# Mount using fuse
ceph-fuse -o keyring=/etc/ceph/ceph.client.admin.keyring, \
    -o client_mount_timeout=60 \
    /mnt/ceph
```

## File Locking

### Advisory Locking

```java
public class FileLocking {
    
    // File locking in Java
    public void lockFile(String path) throws IOException {
        File file = new File(path);
        FileOutputStream fos = new FileOutputStream(file);
        FileLock lock = fos.getChannel().lock();
        
        // Work with file
        // ...
        
        lock.release();
        fos.close();
    }
    
    // Lock with timeout
    public boolean tryLock(String path, long timeout) 
            throws IOException, InterruptedException {
        File file = new File(path);
        FileChannel channel = new FileOutputStream(file)
            .getChannel();
        
        return channel.tryLock(timeout, TimeUnit.MILLISECONDS);
    }
}
```

## File System Monitoring

### Metrics Collection

```java
public class FileSystemMonitor {
    
    // Get file system stats
    public FileSystemStats getStats(String mountPoint) {
        File file = new File(mountPoint);
        
        StatFs stat = new StatFs(file.getPath());
        
        return FileSystemStats.builder()
            .totalBlocks(stat.getBlockCount())
            .freeBlocks(stat.getAvailableBlocks())
            .blockSize(stat.getBlockSize())
            .build();
    }
    
    // Get directory size
    public long getDirectorySize(String path) {
        File directory = new File(path);
        
        return Files.walk(directory.toPath())
            .mapToLong(p -> {
                try {
                    return Files.size(p);
                } catch (IOException e) {
                    return 0L;
                }
            })
            .sum();
    }
}
```

## Best Practices

1. **Use NFSv4**: Better performance and security.

2. **Enable locking**: Prevents data corruption.

3. **Set appropriate rsize/wsize**: Performance tuning.

4. **Use hard mounts with intr**: Handle server failures gracefully.

5. **Monitor capacity**: Alert before disk full.

6. **Backup strategy**: Regular backups of critical files.

7. **Security**: Restrict access with proper permissions.

## Common Mistakes

1. **Not enabling locking**: Concurrent access causes corruption.

2. **Wrong mount options**: Performance issues.

3. **No capacity monitoring**: Disk fills without warning.

4. **Insufficient permissions**: Access denied errors.

5. **No backup**: Data loss from failures.

## Summary

File storage systems remain important for enterprise applications requiring file-based access. Use NFS for Unix/Linux environments, SMB for Windows integration, and distributed file systems (GlusterFS, CephFS) for scalability. Key practices: enable locking, tune mount options, monitor capacity, and implement backup strategies.
