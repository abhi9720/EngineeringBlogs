# Raft & Paxos Consensus Algorithms

## Overview

Consensus algorithms are the backbone of fault-tolerant distributed systems. They enable a group of nodes to agree on a single value despite failures. Paxos has long been the theoretical gold standard, but Raft's understandability-driven design has made it the practical choice for modern systems like etcd and Consul. This guide explores both algorithms and their real-world applications.

## Problem Statement

In a distributed system, nodes can crash, messages can be delayed or lost, and networks can partition. Without consensus, replicas diverge, clients see stale data, and systems lose consistency guarantees. Consensus solves the fundamental problem of getting multiple nodes to agree on the same sequence of operations.

## Raft Consensus Algorithm

### Leader Election

Raft divides time into terms of arbitrary length, each beginning with an election. Servers are in one of three states: Leader, Follower, or Candidate.

```java
enum ServerState {
    FOLLOWER,
    CANDIDATE,
    LEADER
}

public class RaftNode {
    private ServerState state = ServerState.FOLLOWER;
    private int currentTerm = 0;
    private String votedFor = null;
    private int commitIndex = 0;
    private int lastApplied = 0;
    
    // Election timeout (150-300ms random)
    private final ScheduledExecutorService scheduler = Executors.newScheduledThreadPool(1);
    
    public void startElection() {
        state = ServerState.CANDIDATE;
        currentTerm++;
        votedFor = selfId;
        
        // Request votes from all peers
        int votesReceived = 1; // Vote for self
        for (String peer : peers) {
            if (requestVote(peer, currentTerm)) {
                votesReceived++;
            }
        }
        
        if (votesReceived > peers.size() / 2) {
            becomeLeader();
        }
    }
    
    private boolean requestVote(String peer, int term) {
        // RPC call to peer
        VoteResponse response = rpcClient.send(
            new VoteRequest(term, selfId, lastLogIndex, lastLogTerm)
        );
        return response.isVoteGranted();
    }
}
```

### Log Replication

Once a leader is elected, all client requests go through it. The leader appends entries to its log and replicates them to followers.

```java
public class LogReplicator {
    private final RaftNode node;
    private final Map<String, Integer> nextIndex = new ConcurrentHashMap<>();
    private final Map<String, Integer> matchIndex = new ConcurrentHashMap<>();
    
    public void replicateLog(LogEntry entry) {
        // Leader appends to local log
        node.getLog().append(entry);
        
        // Send AppendEntries RPC to all followers in parallel
        for (String follower : node.getPeers()) {
            scheduler.submit(() -> sendAppendEntries(follower));
        }
    }
    
    private void sendAppendEntries(String followerId) {
        int next = nextIndex.get(followerId);
        List<LogEntry> entries = node.getLog().entriesFrom(next);
        
        AppendRequest request = AppendRequest.builder()
            .term(node.getCurrentTerm())
            .leaderId(node.getSelfId())
            .prevLogIndex(next - 1)
            .prevLogTerm(node.getLog().getTerm(next - 1))
            .entries(entries)
            .leaderCommit(node.getCommitIndex())
            .build();
        
        AppendResponse response = rpcClient.appendEntries(followerId, request);
        
        if (response.isSuccess()) {
            matchIndex.put(followerId, next + entries.size() - 1);
            nextIndex.put(followerId, next + entries.size());
            advanceCommitIndex();
        } else {
            // Decrement and retry
            nextIndex.put(followerId, next - 1);
            sendAppendEntries(followerId);
        }
    }
    
    private void advanceCommitIndex() {
        // An entry is committed once a majority of servers have replicated it
        for (int n = node.getCommitIndex() + 1; 
             n <= node.getLog().lastIndex(); n++) {
            int replicas = 1;
            for (int matched : matchIndex.values()) {
                if (matched >= n) replicas++;
            }
            if (replicas > node.getPeers().size() / 2) {
                node.setCommitIndex(n);
                node.applyLogEntries(n);
            }
        }
    }
}
```

### Safety Properties

Raft guarantees five safety properties:
1. **Election Safety**: At most one leader per term
2. **Leader Append-Only**: Leaders never overwrite or delete their log entries
3. **Log Matching**: If two logs contain an entry with the same index and term, they are identical
4. **Leader Completeness**: A committed entry is present in all future leaders' logs
5. **State Machine Safety**: Servers apply log entries in the same order

## Paxos Consensus Algorithm

### Basic Paxos

Paxos operates in three phases using proposers, acceptors, and learners:

```java
public class PaxosNode {
    private long promisedProposalId = -1;
    private long acceptedProposalId = -1;
    private String acceptedValue = null;
    
    // Phase 1: Prepare
    public PrepareResponse prepare(PrepareRequest request) {
        synchronized (this) {
            if (request.getProposalId() > promisedProposalId) {
                promisedProposalId = request.getProposalId();
                return new PrepareResponse(true, acceptedProposalId, acceptedValue);
            }
            return new PrepareResponse(false, acceptedProposalId, acceptedValue);
        }
    }
    
    // Phase 2: Accept
    public AcceptResponse accept(AcceptRequest request) {
        synchronized (this) {
            if (request.getProposalId() >= promisedProposalId) {
                promisedProposalId = request.getProposalId();
                acceptedProposalId = request.getProposalId();
                acceptedValue = request.getValue();
                return new AcceptResponse(true);
            }
            return new AcceptResponse(false);
        }
    }
}
```

### Multi-Paxos

Multi-Paxos optimizes the basic protocol by electing a distinguished leader, eliminating the prepare phase for subsequent instances:

```java
public class MultiPaxos extends PaxosNode {
    private boolean isLeader = false;
    private final List<Long> instanceLog = new ArrayList<>();
    
    public void propose(String value) {
        if (!isLeader) {
            forwardToLeader(value);
            return;
        }
        
        // Skip prepare phase for subsequent instances
        long instanceId = getNextInstanceId();
        
        // Phase 2 only: Accept
        boolean accepted = broadcastAccept(instanceId, value);
        
        if (accepted) {
            instanceLog.add(instanceId);
            learn(instanceId, value);
        }
    }
}
```

## Raft vs Paxos Comparison

| Aspect | Raft | Paxos |
|---|---|---|
| Understandability | High (decomposed into sub-problems) | Low (tricky to implement correctly) |
| Leader Election | Explicit, time-based | Not specified in base protocol |
| Log Replication | Strong leader, sequential commits | Can be optimized with batching |
| Membership Changes | Joint consensus approach | Requires separate mechanism |
| Implementation Complexity | Moderate | High |
| Production Use | etcd, Consul, TiKV | Google Chubby, Spanner |

## Practical Use Cases

### etcd (Raft-based)

```java
@Service
public class EtcdDistributedConfig {
    
    private final Client etcdClient;
    
    public String getConfigValue(String key) {
        GetResponse response = etcdClient.get(ByteSequence.from(key, UTF_8));
        if (response.getKvs().isEmpty()) {
            return null;
        }
        return response.getKvs().get(0).getValue().toString(UTF_8);
    }
    
    public void watchConfigChanges(String key, Consumer<String> onChange) {
        etcdClient.watch(ByteSequence.from(key, UTF_8))
            .listen(event -> {
                for (Event e : event.getEvents()) {
                    String newValue = e.getKv().getValue().toString(UTF_8);
                    onChange.accept(newValue);
                }
            });
    }
}
```

### ZooKeeper (Zab protocol, Paxos-inspired)

```java
public class ZooKeeperLeaderElection {
    private final ZooKeeper zk;
    private String leaderPath;
    
    public void electLeader(String candidateId) throws Exception {
        String candidatePath = zk.create(
            "/election/candidate-",
            candidateId.getBytes(),
            ZooDefs.Ids.OPEN_ACL_UNSAFE,
            CreateMode.EPHEMERAL_SEQUENTIAL
        );
        
        // Watch the preceding node
        List<String> children = zk.getChildren("/election", false);
        Collections.sort(children);
        
        int index = children.indexOf(candidatePath.substring("/election/".length()));
        if (index == 0) {
            // I am the leader
            becomeLeader();
        } else {
            // Watch the node before me
            String watchedNode = "/election/" + children.get(index - 1);
            zk.exists(watchedNode, event -> {
                if (event.getType() == Watcher.Event.EventType.NodeDeleted) {
                    electLeader(candidateId); // Re-elect
                }
            });
        }
    }
}
```

## Best Practices

- Start with Raft for new projects; it's easier to implement correctly and debug
- Use strict time bounds for election timeouts to prevent split-brain scenarios
- Implement pre-vote checks to avoid disrupting an established leader during network partitions
- Always persist log entries to disk before sending responses to prevent data loss on crash
- Use joint consensus for membership changes to ensure availability during reconfiguration
- Monitor leader stability metrics and track election rate as a key health indicator

## Common Mistakes

- Ignoring disk I/O latency in leader election timeout configuration, causing cascading elections
- Not handling stale leaders that haven't heard about a new term (requires epoch-based fencing)
- Underestimating the complexity of snapshotting and log compaction (trickiest part of production Raft)
- Using asynchronous replication without proper commit index tracking
- Failing to implement proper backoff for election timeouts, causing thundering herd problems

## Summary

Raft and Paxos solve the fundamental problem of distributed consensus but with different design philosophies. Raft's emphasis on understandability has made it the go-to choice for most modern systems, while Paxos remains important for understanding the theoretical foundations. The practical implementations in etcd, Consul, and ZooKeeper demonstrate that careful attention to log replication, leader election, and membership management is essential for production-grade consensus systems.
